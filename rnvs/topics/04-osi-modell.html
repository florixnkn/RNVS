Worum geht's?

Damit Computer weltweit miteinander kommunizieren können, brauchen wir ein gemeinsames Regelwerk — ein Referenzmodell. Das OSI-Modell teilt diese Aufgabe in 7 sauber getrennte Schichten, das Internet-Modell komprimiert das Ganze auf 4 Schichten. Beide erklären, wie Daten von einer App über Kabel und Router bis zur App auf der Gegenseite wandern.

Außerdem lernst du, was PDU, SDU und PCI bedeuten — die Grundvokabeln für jeden Klausur-Aufgabentyp rund ums Schichtenmodell.

Stell dir vor …

… du schickst einen Brief. Sieben „Abteilungen" arbeiten nacheinander daran: (7) Du schreibst den Brief · (6) Er wird formatiert und in Kurzschrift übersetzt · (5) Ein Termin für die Übergabe wird vereinbart · (4) Der Brief wird in einen Umschlag mit Absender/Empfänger gesteckt · (3) Die Postroute wird geplant · (2) Ein lokaler Postbote nimmt ihn mit · (1) Der Brief wird als physisches Signal (Licht im Glasfaser-Kabel) übertragen. Jede Abteilung kennt nur ihren eigenen Job — genau so funktionieren die 7 OSI-Schichten. Auf der Empfängerseite läuft der Prozess rückwärts ab: jede Schicht packt ihr eigenes „Etikett" wieder ab, bis oben der Inhalt ankommt.

Theorie & Konzepte

Das OSI-Referenzmodell (7 Schichten)

Das OSI-Modell (Open Systems Interconnection) wurde von der ISO entwickelt, um Netzwerkstandards herstellerunabhängig zu definieren. Es zerlegt die Netzwerkkommunikation in 7 Schichten, jede mit klarer Aufgabe, eigenem PDU-Namen und eigenen Protokollen.

Das Internet-Referenzmodell (4 Schichten)

Das Internet-Modell (auch TCP/IP-Modell genannt) ist schlanker als OSI und spiegelt besser wider, wie das Internet tatsächlich gebaut ist. Es fasst die OSI-Schichten 5, 6 und 7 zur „Anwendung" zusammen und die Schichten 1 und 2 zum „Netzanschluss".

OSI vs. Internet – Anwendungsschicht

Im OSI-Modell sind Sitzung (5), Darstellung (6) und Anwendung (7) getrennte Schichten. Im Internet-Modell macht das alles die Anwendung selbst — z. B. ist TLS die Verschlüsselung direkt in der App (kein gesonderter OSI-Layer 6), und JSON ist das Format direkt in der App (kein gesonderter OSI-Layer 6). In der Praxis sind Schichten 5 und 6 bei OSI oft leer.

PDU, SDU, PCI, ICI und IDU

Jede Schicht verpackt die Daten der darüberliegenden Schicht in ihr eigenes „Paket" — dafür brauchen wir präzise Begriffe:

  • SDU = Service Data Unit = die Nutzdaten, die von oben reinkommen (der Inhalt des Briefes, z. B. der HTTP-Body)
  • PCI = Protocol Control Information = der Header, den diese Schicht selbst draufschreibt (der Aufkleber auf dem Umschlag: Absender, Empfänger, Porto)
  • PDU = Protocol Data Unit = SDU + PCI = das fertige Paket dieser Schicht (der versiegelte Umschlag, z. B. das TCP-Segment)
  • ICI = Interface Control Information = interne Steuerinfo zwischen benachbarten Schichten, die NICHT im Netz übertragen wird (mündliche Anweisung an den Postboten: „Express!")
  • IDU = Interface Data Unit = SDU + ICI bei der Übergabe zwischen Schichten
Die Grundformeln
(N)-PDU = (N)-PCI + (N)-SDU (N)-SDU = (N+1)-PDU

N steht für die Schicht-Nummer (z. B. 4 = Transport). Die PDU der Schicht N+1 (z. B. HTTP-Nachricht) wird zur SDU der Schicht N (z. B. TCP). TCP fügt seinen Header (PCI) hinzu und erzeugt sein eigenes Segment (PDU). Dieses Segment wird dann wiederum zur SDU der Schicht 3 (IP), die ihren Header draufpackt — und so weiter.

Peer-Kommunikation

Gleiche Schichten auf beiden Seiten einer Verbindung heißen Peer-Entities. Sie kommunizieren logisch miteinander — z. B. redet TCP auf Sender-Seite „logisch" mit TCP auf Empfänger-Seite. Physisch läuft jedes Bit aber komplett durch alle Schichten nach unten, über die Leitung und auf der anderen Seite durch alle Schichten wieder hoch.

Logisch vs. Physisch

Logisch: HTTP auf Seite A „glaubt", direkt mit HTTP auf Seite B zu reden — als hätten sie eine unsichtbare direkte Verbindung. Das ist die Peer-Kommunikation.
Physisch: Die HTTP-Nachricht geht zuerst runter zu TCP, dann IP, dann Ethernet, dann über Kupferkabel / Glasfaser, beim Router durch die Vermittlungsschicht, dann weiter bis zum Empfänger, wo alles wieder hochgereicht wird. Es gibt keine echte Abkürzung.

Vor- und Nachteile der Schichtenarchitektur & OSI-Kritik

ModularitätSchichten einzeln austauschbar (z. B. IP v4 → v6)
WiederverwendungTCP kann von HTTP, FTP, SMTP etc. genutzt werden
Komplexitätsreduktion1 klar definiertes Problem pro Schicht
StandardisierungKlare Schnittstellen, Interoperabilität weltweit
Nachteile & Overhead

Jede Schicht fügt einen Header hinzu → Overhead steigt pro Schicht. Performance-Verlust durch viele Übergaben. Cross-Layer-Optimierung ist schwierig (eine Schicht darf die andere nicht einfach umgehen). Redundanz: manche Funktionen (z. B. Fehlererkennung) werden in mehreren Schichten wiederholt.

OSI-Kritik: Die OSI-Schichten 5 (Sitzung) und 6 (Darstellung) sind in der Praxis oft leer — das Internet hat sie nie gebraucht. Der OSI-Stack hat sich als Protokoll-Familie nie durchgesetzt; TCP/IP war bereits verbreitet und einfacher zu implementieren. Heute ist OSI vor allem ein pädagogisches Werkzeug, das Konzepte (Schichten, PDU/SDU, Peer-Kommunikation) präzise beschreibt.

Schritt für Schritt: PDU-Aufbau analysieren

PDU einer Schicht bestimmen

  1. Identifiziere die Schicht N — z. B. TCP = Schicht 4, IP = Schicht 3.
  2. Was ist die (N)-SDU? → Das, was von Schicht N+1 reinkommt — z. B. bei TCP sind das die HTTP-Daten (= HTTP-PDU = Schicht-7-PDU).
  3. Was ist die (N)-PCI? → Alle Felder, die das Protokoll selbst hinzufügt — z. B. bei TCP: Source-Port, Dest-Port, Sequence Number, Flags, Checksum (= TCP-Header).
  4. Was ist die (N)-PDU? → PCI + SDU = das fertige Paket dieser Schicht — z. B. das komplette TCP-Segment.
  5. Was bekommt Schicht N-1? → Die (N)-PDU wird zur (N-1)-SDU, dann kommt noch (N-1)-PCI dazu → neue PDU auf Schicht N-1.
Durchgerechnetes Beispiel: HTTP über TCP über IP
Schicht 7: HTTP-Daten = 200 Byte (SDU für Schicht 4) Schicht 4: TCP-Header (PCI) = 20 Byte → TCP-Segment (PDU) = 220 Byte Schicht 3: IP-Header (PCI) = 20 Byte → IP-Paket (PDU) = 240 Byte

Das TCP-Segment (220 Byte) wird zur SDU von Schicht 3. IP fügt 20 Byte Header hinzu → IP-Paket = 240 Byte. Das IP-Paket wird dann zur SDU von Schicht 2 (Ethernet), die ihren Frame-Header und Frame-Trailer hinzufügt.

Schnittbildung bestimmen

  1. Sind physische Geräte eingezeichnet? — Router, Switches, Hosts mit Kabelverbindungen → Systemschnitt
  2. Ist das Netz als Wolke oder Blob dargestellt? — Internet-Symbol ohne Details → Dienstschnitt
  3. Sind Protokoll-Stacks (Schichten-Boxen) pro Knoten gezeigt? — HTTP/TCP/IP-Layer nebeneinander für verschiedene Systeme → Protokollschnitt
  4. Begründung formulieren: „Hier sind [X] dargestellt, daher handelt es sich um den [Typ]-Schnitt."

OSI-Modell auswendig lernen

  1. Merkspruch wählen: „Alle Priester Saufen Tequila Vor Sieben Bleibender" — von oben nach unten: 7 → 1.
  2. Richtung merken: Von oben (7 = Anwendung / Nutzer) nach unten (1 = Kabel / Physik).
  3. Pro Schicht merken: 1 Aufgabe + 1–2 Protokolle — z. B. Schicht 3 = Routing → IP.
  4. PDU-Namen: Message (7–5) · Segment/Datagram (4) · Paket (3) · Frame (2) · Bits (1).

Typische Fallen

❌ FalschTransport ist OSI-Schicht 3.
✅ RichtigTransport ist Schicht 4. Vermittlung/IP ist Schicht 3. Merke: TCP=4, IP=3.
❌ FalschDie SDU enthält den Header (PCI).
✅ RichtigPDU = PCI (Header) + SDU (Nutzdaten). Die SDU ist das, was von oben reinkommt — also ohne eigenen Header. Der Header kommt erst durch die aktuelle Schicht dazu.
❌ FalschICI ist der Header, der im Netz übertragen wird.
✅ RichtigICI bleibt im Host — es ist vertikale Steuerinfo zwischen benachbarten Schichten. Im Netz steckt der PCI. Merke: PCI = über-Netz, ICI = im-Host.
❌ FalschDas Internet-Modell hat 5 Schichten.
✅ RichtigLMU nutzt 4 Schichten (Anwendung, Transport, Vermittlung, Netzanschluss). Manche Bücher trennen Physical und Data Link — LMU fasst beides als „Netzanschluss" zusammen.
❌ FalschIm Internet-Modell gibt es eine Darstellungsschicht für TLS/Verschlüsselung.
✅ RichtigDas Internet-Modell hat KEINE Darstellungsschicht. TLS, JSON, Codierung — das macht die Anwendung selbst. Es gibt nur 4 Schichten.
❌ FalschPeer-Kommunikation bedeutet, dass zwei gleiche Schichten direkt physisch verbunden sind.
✅ RichtigPeer-Kommunikation ist logisch. Physisch läuft jede Nachricht komplett nach unten, über die Leitung, und auf der anderen Seite wieder hoch — keine direkte Abkürzung.

🧩 Wissenscheck

Teste dich selbst! Tippe deine Antwort ein oder wähle eine Option — du bekommst sofort Rückmeldung. Mit „Lösung" siehst du die Erklärung.

1.Auf welcher OSI-Schicht (Nummer) arbeitet TCP?

Schicht
TCP = Transport = Schicht 4. (IP liegt auf Schicht 3, Vermittlung.)

2.Welche OSI-Schichten fasst das Internet-Modell zur einen „Anwendungs"-Schicht zusammen?

Die OSI-Schichten 5 (Sitzung), 6 (Darstellung) und 7 (Anwendung) werden im Internet-Modell zu einer einzigen Anwendungsschicht.

3.Richtig oder falsch: „Die SDU enthält bereits den Header (PCI)."

Falsch — die SDU sind die reinen Nutzdaten von oben. Erst PDU = PCI (Header) + SDU.

4.🎓 Klausur 2017: Wie entsteht aus einer Service Data Unit (SDU) eine Protocol Data Unit (PDU)?

Jede Schicht hängt ihre Protokoll-Steuerinformation (PCI, der „Header") vor die SDU. Wörtliche Klausurfrage 2017/2018.

5.🎓 Klausur 2017: Eine Protokollinstanz der Schicht N tauscht PDUs mit ihrer Peer-Entity aus (die liegt ebenfalls auf Schicht N). An welche Schicht wird die PDU zur Übertragung übergeben?

An Schicht N−1 — die PDU der Schicht N wird zur SDU der Schicht N−1. Die Peer-Entity liegt auf derselben Schicht N, kommuniziert wird aber immer über die darunterliegende Schicht.

6.🎓 Klausurklassiker (2014/2017): Welches dieser Protokolle liegt NICHT auf der Anwendungsschicht?

ARP gehört zur Sicherungsschicht (2). Merke fürs Einordnen: HTTP(S), IMAP, POP3, SMTP, DNS, FTP → Anwendung (7) · TCP, UDP → Transport (4) · ICMP, OSPF, IP → Vermittlung (3) · ARP, PPP → Sicherung (2).
