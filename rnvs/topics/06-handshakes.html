Worum geht's?

Bevor zwei Computer Daten austauschen, müssen sie sich absprechen: Wollen wir überhaupt miteinander reden? Und: Haben wir beide verstanden, dass wir jetzt reden? Der Handshake ist genau dieses gegenseitige Einverständnis — ein kurzes Begrüßungsritual, bevor die eigentlichen Daten fließen.

Stell dir vor …

… du rufst jemanden an und sagst einfach los, ohne zu warten, ob die andere Seite abgenommen hat. Du könntest die ganze Zeit in eine Mailbox sprechen und es nicht merken. Genau das ist das Problem ohne Handshake! Beim 2-Wege-Handshake wartest du, bis der andere „Hallo" sagt. Beim 3-Wege-Handshake sagst du dann noch einmal kurz „Alles klar, ich höre dich" — und erst dann redet ihr wirklich. TCP macht genau das: SYN, SYN-ACK, ACK — und dann fließen die Daten.

Es gibt zwei grundlegend verschiedene Arten, wie Computer Daten schicken können. Der Unterschied: ob sie vorher eine „Leitung aufbauen" oder einfach drauflosschicken.

Der 2-Wege-Handschlag und sein Problem

Beim 2-Wege-Handschlag schickt der Client einen Verbindungswunsch (CONN_REQ), und der Server antwortet mit einer Bestätigung (CONN_ACK). Klingt einfach — aber es gibt ein heimtückisches Problem:

Achtung: Das Duplikat-Problem

Noch tückischer: Ein verspätetes Duplikat einer alten CONN_REQ kann ankommen, wenn die ursprüngliche Verbindung längst beendet ist. Der Server hält diese für eine neue Verbindungsanfrage und reserviert Ressourcen — für eine Verbindung, die der Client gar nicht mehr will. Ergebnis: wieder eine halboffene Verbindung auf dem Server.

Der 3-Wege-Handschlag: die Lösung

Mit einem dritten Schritt — dem finalen ACK vom Client — bestätigt der Client, dass er die Antwort des Servers empfangen hat. Erst dann geht der Server in den Zustand „Verbunden". So kann keine halboffene Verbindung durch Verlust der zweiten Nachricht entstehen.

Merksatz: Warum 3?

Beim 2-Wege-Handschlag kann der Empfänger der zweiten Nachricht nicht wissen, ob seine Antwort ankam. Mit einem dritten Schritt (ACK) bestätigt der Client genau das. Der Server geht erst dann in „Verbunden" — keine halboffene Verbindung mehr.

Zustandsdiagramme: Client & Server

Ein Zustandsdiagramm zeigt nicht Nachrichten über die Zeit (das wäre ein Sequenzdiagramm), sondern die Zustände, in denen sich eine Seite befinden kann, und die Übergänge zwischen ihnen. Jeder Übergang hat die Form Trigger / Aktion.

Verbindungsabbau

Auch das Ende einer Verbindung wird ordentlich ausgehandelt. Beim vereinfachten 3-Wege-Abbau schickt der Client ein FIN, der Server antwortet mit ACK + FIN, und der Client bestätigt mit einem letzten ACK. Im echten TCP wird jede Richtung separat geschlossen (4-Wege-Abbau: FIN, ACK, FIN, ACK).

Das Two-Generals-Problem

Kein Handshake kann mathematisch garantieren, dass beide Seiten im gleichen Zustand sind. Der Grund: die letzte Nachricht kann immer verloren gehen — und der Sender weiß nicht, ob sie ankam. Man nennt das das Two-Generals-Problem: Zwei Armeen auf zwei Hügeln müssen sich auf einen Angriffszeitpunkt einigen. Jeder Bote kann abgefangen werden. Egal wie viele Boten man schickt — es gibt keinen Beweis, dass der letzte ankam.

Trotzdem ist der 3-Wege-Handschlag in der Praxis ausreichend, weil Verluste selten sind, Timeouts und Wiederholungen fast alle Probleme lösen, und die Anwendungsschicht zusätzlich absichern kann.

Handshake Schritt für Schritt

So gehst du vor, wenn du in der Klausur ein Zustandsdiagramm oder eine Fehlerszenario-Analyse zeichnen sollst:

Zustandsdiagramm zeichnen

  1. Zustände identifizieren: Für den Client z. B. Idle → SYN-Sent → Verbunden → Closed. Für den Server: Listen → SYN-Received → Verbunden → Closed.
  2. Übergänge bestimmen: Jeder Übergang hat die Form Trigger / Aktion. Beispiel: SYN-ACK empfangen / sende ACK.
  3. Diagramm zeichnen: Startzustand mit ausgefülltem Kreis ●, Zustände als Oval/Kreis, Endzustand mit Doppelkreis ◉. Pfeile beschriften.
  4. Beide Seiten getrennt: Client und Server haben je ein eigenes Zustandsdiagramm — nicht vermischen!

Fehlerszenario analysieren

  1. Welche Nachricht geht verloren? Im Sequenzdiagramm: welcher Pfeil wird unterbrochen?
  2. In welchem Zustand ist jede Seite nach dem Verlust? Schreibe für Client und Server den aktuellen Zustand auf.
  3. Gibt es eine Inkonsistenz? Wenn Client und Server in verschiedenen Zuständen sind → halboffene Verbindung.
  4. Wie wird das gelöst? Timeout beim wartenden Client → CONN_REQ erneut senden. Server: abhängig vom Protokoll entweder Verbindung neu akzeptieren oder bestehende verwenden.
Übergangs-Notation
Trigger / Aktion

Trigger = was empfangen wurde (auslösendes Ereignis). Aktion = was daraufhin gesendet wird. Ein Strich — bedeutet „keine Aktion". Beispiel: ACK empfangen / — heißt: ACK kommt an, nichts wird zurückgesendet, Zustandswechsel trotzdem.

Typische Fallen

❌ FalschVerbindungsorientiert bedeutet immer TCP.
✅ RichtigVerbindungsorientiert ist ein Konzept — TCP ist nur eine Implementierung. Auch ATM, X.25 und andere Protokolle sind verbindungsorientiert.
❌ FalschVerbindungslos ist dasselbe wie unzuverlässig.
✅ RichtigVerbindungslos und Zuverlässigkeit sind orthogonal — ein verbindungsloses Protokoll könnte Zuverlässigkeit bieten (z. B. QUIC). Die meisten tun es aus Performance-Gründen nicht.
❌ FalschDer 2-Wege-Handschlag ist sicher, weil der Server antwortet.
✅ RichtigDer Server weiß nicht, ob seine CONN_ACK ankam. Geht sie verloren, ist der Server „verbunden", der Client wartet noch → halboffene Verbindung. Erst der 3. Schritt löst das.
❌ FalschDer 3-Wege-Handschlag garantiert eine fehlerfreie Verbindung.
✅ RichtigDas Two-Generals-Problem beweist: kein Handshake gibt eine 100%-Garantie. Die letzte Nachricht kann immer verloren gehen. 3-Wege ist nur in der Praxis ausreichend.
❌ FalschZustandsdiagramm und Sequenzdiagramm sind dasselbe.
✅ RichtigEin Sequenzdiagramm zeigt Nachrichten über die Zeit (Pfeile zwischen zwei Seiten). Ein Zustandsdiagramm zeigt Zustände + Übergänge einer Seite (Kreise + Pfeile mit Trigger/Aktion). Immer genau lesen, was gefragt ist!
❌ FalschSYN ist eine eigene Nachricht (ein eigenes Paket).
✅ RichtigSYN ist ein Flag (Bit) in einer TCP-PDU — das Segment kann zusätzlich Daten enthalten. SYN ist kein eigenständiger Nachrichtentyp, sondern ein Steuerbit.

🧩 Wissenscheck

Teste dich selbst! Tippe deine Antwort ein oder wähle eine Option — du bekommst sofort Rückmeldung. Mit „Lösung" siehst du die Erklärung.

1.Aus wie vielen Nachrichten besteht der TCP-Verbindungsaufbau (der „Handshake")?

Nachrichten
3 — SYN, SYN-ACK, ACK (der 3-Wege-Handschlag).

2.Richtig oder falsch: „Ein 2-Wege-Handshake garantiert eine sichere, konsistente Verbindung."

Falsch — geht die 2. Nachricht verloren, entsteht eine halboffene Verbindung. Deshalb nutzt TCP den 3-Wege-Handschlag.

3.Welches Prinzip beweist, dass KEIN Handshake 100 % Konsens garantieren kann?

Das Two Generals' Problem: Die letzte Nachricht kann immer verloren gehen, ihr Absender weiß nie sicher, ob sie ankam.

4.🎓 Klausur 2022: Unter welchen Bedingungen ist verbindungslose Kommunikation von Vorteil?

Die Musterlösung nennt: sehr kurze Kommunikationen (DNS, Ping), stark schwankendes Datenaufkommen und Kommunikation ohne Schaden bei Verlust. Für Zuverlässigkeit und Reihenfolge braucht man dagegen eine verbindungsorientierte Verbindung.

5.🎓 Klausur 2009/2014: Welche dieser Dienste sind typisch verbindungsorientiert?

Verbindungsorientiert: Telefon, ISDN, Mobilfunk (ohne VoIP), Standleitung — erst Verbindungsaufbau, dann exklusive Nutzung. Verbindungslos: IP, Post, Ethernet, Broadcasting — jedes Paket trägt seine Zieladresse selbst.

6.🎓 Klausur 2009: Richtig oder falsch: „Bei verbindungsorientierter Kommunikation brauchen die Nutzdaten nach dem Verbindungsaufbau keine Zielinformationen mehr."

Richtig — zuerst werden Steuerpakete zum Verbindungsaufbau geschickt, danach fließen die Nutzdaten durch die stehende Verbindung. Bei verbindungsloser Kommunikation muss dagegen jedes Paket mit der Zieladresse versehen werden.
