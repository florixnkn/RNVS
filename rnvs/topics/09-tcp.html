Worum geht's?

TCP (Transmission Control Protocol) ist das Rückgrat des Internets: HTTP, E-Mail, SSH – alles läuft über TCP. Es sorgt dafür, dass Daten zuverlässig, vollständig und in der richtigen Reihenfolge ankommen. Dafür baut TCP vor dem ersten Datenbyte eine Verbindung auf, nummeriert jedes Byte und verlangt eine Quittung vom Empfänger.

Stell dir vor …

… du schickst einen wichtigen Roman per Post – aber nicht das ganze Buch auf einmal, sondern Seite für Seite in nummerierten Umschlägen. Der Empfänger schickt dir nach jeder empfangenen Seite eine Postkarte zurück: „Seite 47 angekommen, warte auf Seite 48." Kommt mal eine Seite nicht an, sendest du sie nochmal. Genau das macht TCP – nur dass statt Seiten einzelne Bytes nummeriert werden, und die „Postkarte" eine ACK-Nummer trägt.

TCP-Header & Flags

Jedes TCP-Segment beginnt mit einem Header von mindestens 20 Byte. Dort stehen alle Steuerinformationen: Ports, Sequenznummern, Flags und das Empfangsfenster. Erst danach kommen die eigentlichen Nutzdaten.

Die wichtigsten TCP-Flags

Flags sind Schalter im Header – jedes ist genau 1 Bit. Mehrere können gleichzeitig gesetzt sein (z.B. SYN+ACK).

Merksatz: SYN und FIN kosten je eine SeqNr

Auch wenn SYN und FIN keine Nutzdaten enthalten, verbrauchen sie trotzdem je einen Sequenznummer-Slot. Deshalb ist nach einem SYN mit SeqNr=6000 die erste AckNr 6001 – nicht 6000!

Sequenznummer (SeqNr) vs. Bestätigungsnummer (AckNr)

Das sind die zwei häufigsten Klausur-Fragen – und die häufigsten Fehlerquellen. Lerne den Unterschied auswendig:

Formel: SeqNr und AckNr
AckNr = SeqNr + Segmentlänge (in Byte)

SeqNr = Byte-Nummer des 1. Daten-Bytes im Segment
AckNr = SeqNr des nächsten erwarteten Bytes (= alles bis AckNr−1 ist bestätigt)
Beispiel: SeqNr=127, 80 Byte → Bytes 127–206 → AckNr=207

Kumulative Quittungen und das Lücken-Problem

Standard-TCP quittiert kumulativ: die AckNr gibt das nächste erwartete Byte an und bestätigt damit alle Bytes darunter. Das Problem: kommt ein Segment in der Mitte verloren, kann der Empfänger die danach empfangenen Bytes nicht bestätigen.

Achtung: Lücken-Problem bei kumulativem ACK

Bytes 1–500 ankommen ✓, Bytes 501–1000 verloren ✗, Bytes 1001–2000 ankommen ✓. Der Empfänger kann nur ACK 501 schicken – die schon empfangenen Bytes 1001–2000 bleiben „unsichtbar" für den Sender. Ohne SACK muss der Sender alles ab 501 erneut schicken (Go-Back-N).

Selektive Quittungen (SACK) – RFC 2018

SACK löst das Lücken-Problem: Der Empfänger sendet zusätzlich zur normalen AckNr eine Liste der bereits empfangenen Byte-Bereiche. Der Sender kann dann gezielt nur die fehlenden Segmente wiederholen – kein unnötiger Re-Send.

Handshake, Datenübertragung & Abbau

3-Wege-Handshake (Verbindungsaufbau)

Bevor TCP auch nur ein Datenbyte sendet, einigen sich Client und Server über drei Schritte auf je eine Initial Sequence Number (ISN) – eine zufällig gewählte Startnummer für jede Richtung. Das erhöht die Sicherheit (Anti-Spoofing) und vermeidet Verwechslungen mit alten Paketen.

  1. SYN (Client → Server): Client wählt eine zufällige ISN (hier: 6000) und schickt ein Segment mit gesetztem SYN-Flag. Noch keine Nutzdaten.
  2. SYN+ACK (Server → Client): Server bestätigt (AckNr = 6001 = ISNc + 1) und sendet gleichzeitig seine eigene ISN (9000) mit einem SYN-Flag.
  3. ACK (Client → Server): Client bestätigt die Server-ISN (AckNr = 9001 = ISNs + 1). Danach sind beide Seiten ESTABLISHED und Daten können fließen.

Datenübertragung: HTTP-Beispiel (Blatt 6)

Nach dem Handshake sendet der Client seinen HTTP-Request (50 Byte). Die Sequenznummern setzen sich dabei direkt fort – da der Client nach dem Handshake bei SeqNr=6001 war und 50 Byte sendet, bestätigt der Server mit AckNr=6051.

HTTP-Antwortzeit: TCP vs. UDP

Setup: RTT = 300 ms (Netzverzögerung 150 ms × 2), Request 50 Byte, Response 1000 Byte.

TCP: 3-Way-Handshake = 1,5 RTT, dann Request+Response = 1 RTT → Antwort beim Client nach 2 RTT = 600 ms.

UDP: Kein Handshake nötig. Request raus, Response zurück = 2 × 150 ms = 300 ms. TCP braucht doppelt so lang!

4-Wege-Verbindungsabbau (FIN)

TCP schließt jede Richtung separat. Der erste FIN beendet nur den Datenfluss in einer Richtung – die andere Richtung kann danach noch Daten senden (Halbschluss). Erst wenn beide FINs quittiert sind, ist die Verbindung wirklich zu.

Port-Tausch bei ACKs

Gut zu wissen: Ports werden gespiegelt

Sendet Host A ein Segment mit Quellport 302 → Zielport 80, dann dreht Host B im ACK die Ports um: Quellport 80 → Zielport 302. Sender und Empfänger tauschen die Rollen – und damit auch die Ports.

Typische Fallen

❌ FalschDie Sequenznummer zählt, wie viele Segmente schon gesendet wurden.
✅ RichtigSeqNr = Byte-Nummer des ERSTEN Daten-Bytes in diesem Segment. Segmente interessieren gar nicht — nur Bytes!
❌ FalschAckNr = Nummer des zuletzt empfangenen Bytes.
✅ RichtigAckNr = Nummer des NÄCHSTEN erwarteten Bytes. Wenn Byte 206 das letzte empfangene war, dann AckNr = 207. Merksatz: AckNr ist exklusiv – „ich warte auf …".
❌ FalschSYN verbraucht keine Sequenznummer, weil es ja keine Nutzdaten trägt.
✅ RichtigSYN belegt 1 SeqNr-Slot – deshalb ist die erste AckNr nach SYN(6000) immer 6001. Gleiches gilt für FIN!
❌ FalschSACK ersetzt die normale ACK-Nummer.
✅ RichtigSACK ist ZUSÄTZLICH zur normalen AckNr (im Options-Feld). Die AckNr bleibt weiterhin der höchste lückenlos empfangene Byte+1.
❌ FalschTCP-Header = 8 Byte.
✅ Richtig8 Byte ist der UDP-Header. TCP hat mindestens 20 Byte, mit Options (z.B. SACK) bis zu 60 Byte.
❌ FalschDie Ports im ACK sind dieselben wie im Original-Segment.
✅ RichtigIm ACK werden Quell- und Zielport gespiegelt. A→B(302→80) wird im ACK zu B→A(80→302).
❌ FalschTCP baut die Verbindung mit 2 Paketen auf (SYN + SYN-ACK).
✅ RichtigEs sind 3 Pakete (3-Wege-Handshake): SYN → SYN+ACK → ACK. Erst das dritte Paket schließt den Handshake ab.

Typische Klausur-Aufgabentypen

  1. SeqNr/AckNr berechnen: Start-SeqNr + Segmentlänge = AckNr des Empfängers.
  2. Sequenzdiagramm zeichnen: 3-Wege-HS mit korrekten Nummern; Daten + ACK; Abbau.
  3. Out-of-Order: Seg 2 vor Seg 1 → AckNr bleibt bei erwartetem Seg 1 (Duplikat-ACK).
  4. ACK-Verlust + Retransmit: Seg 1 ACK verloren → Timeout → Seg 1 erneut; Empfänger ignoriert Duplikat, sendet aktuelles ACK.
  5. SACK-Blöcke berechnen: Liste empfangener Bereiche außerhalb der Lücken.
  6. HTTP-Antwortzeit: 2 RTT für TCP, 1 RTT für UDP (kein Handshake).

🧩 Wissenscheck

Teste dich selbst! Tippe deine Antwort ein oder wähle eine Option — du bekommst sofort Rückmeldung. Mit „Lösung" siehst du die Erklärung.

1.Ein Segment hat SeqNr = 127 und 80 Byte Daten (also Bytes 127–206). Welche AckNr sendet der Empfänger zurück?

Die AckNr ist das nächste erwartete Byte: nach Byte 206 kommt 207.

2.Wie viele Byte hat der TCP-Header mindestens?

Byte
20 Byte (bis 60 Byte mit Options). Achtung: 8 Byte wäre UDP!

3.Was bedeutet die Acknowledgment Number (AckNr)?

Die AckNr nennt das nächste erwartete Byte und bestätigt damit kumulativ alles davor.

4.🎓 Klausur 2014/2019: Der Server sendet Segment S1 mit gesetztem FIN,ACK-Flag, 0 Byte Nutzdaten und Seq = 2051. Welche AckNr sendet der Client als Antwort zurück?

2052 — obwohl S1 keine Nutzdaten enthält! Bei gesetztem SYN- oder FIN-Flag wird ein „Phantom-Byte" mitgezählt: AckNr = Seq + Nutzdaten + 1. Klassische Falle in den Klausuren 2014, 2015 und 2019.

5.🎓 Klausur 2019: Welche gesetzten Flags im Header eines TCP-Segments werden immer durch ein ACK quittiert?

SYN (Verbindungsaufbau) und FIN (Verbindungsabbau) müssen immer quittiert werden — deshalb zählen sie auch je ein Phantom-Byte in der Sequenznummer.

6.🎓 Klausur 2009: Im TCP-Zustandsdiagramm des Clients: In welchem Zustand befindet sich der Client, nachdem er das erste SYN gesendet hat?

Ablauf beim Client: CLOSED → SYN senden → SYN_SENT → SYNACK empfangen, ACK senden → ESTABLISHED. Beim Abbau: FIN senden → FIN_WAIT_1 → FIN_WAIT_2 → (FIN empfangen, ACK senden) → TIME_WAIT → warten → CLOSED.
