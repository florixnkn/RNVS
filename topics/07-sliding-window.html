Worum geht's?

Wenn Pakete im Netz verloren gehen, vertauscht ankommen oder doppelt auftauchen – wie merkt der Empfänger das überhaupt? Die Antwort: Sequenznummern. Und damit der Sender nicht nach jeder Nachricht untätig wartet, gibt es das Sliding Window – ein rollierendes Sendefenster, das die Leitung maximal auslastet.

Stell dir vor …

… du schickst einem Freund einen langen Roman per Post – Seite für Seite in separaten Umschlägen. Damit er merkt, ob ein Brief fehlt oder die Reihenfolge stimmt, nimmst du auf jeden Umschlag eine Nummer: 1, 2, 3 … Das ist die Sequenznummer.

Beim normalen Briefeschreiben wartest du auf eine Antwort, bevor du den nächsten Brief abschickst – das wäre Stop-and-Wait. Sehr langsam! Wenn du aber schon die nächsten drei Seiten einwirfst, bevor die erste Antwort zurückkommt, nutzt du das Sliding Window. Dein „Fenster" – die Anzahl Briefe, die gleichzeitig unterwegs sein dürfen – gleitet mit jeder Bestätigung weiter.

Sequenznummern erkennen

Mit Sequenznummern erkennt der Empfänger sofort: Duplikat (gleiche Nummer zweimal), Reihenfolge-Fehler (kleinere Nummer nach größerer) und Verlust (eine Nummer fehlt in der Sequenz).

Theorie: Stop-and-Wait & Sliding Window

Stop-and-Wait – der brave, langsame Weg

Beim Stop-and-Wait-Verfahren sendet der Sender genau eine Nachricht (PDU) und wartet, bis er eine Quittung (ACK) empfangen hat. Erst dann darf die nächste Nachricht raus. Fenstergröße w = 1.

Fehlerfälle: Was passiert, wenn etwas schiefläuft?

Sliding Window – die Schnelle-Post-Lösung

Mit Fenstergröße w > 1 darf der Sender mehrere Pakete auf einmal „in Umlauf" bringen, ohne auf ACKs zu warten. Sobald das erste ACK zurückkommt, rutscht das Fenster um eine Position nach rechts – das nächste Paket kann sofort raus.

Go-Back-N: Was passiert bei einem Verlust im Fenster?

Im Go-Back-N-Verfahren akzeptiert der Empfänger Nachrichten nur strikt in Reihenfolge. Fehlt eine Nachricht, werden alle folgenden verworfen. Der Sender wiederholt ab der verlorenen Nachricht alles.

Go-Back-N – Achtung!

N2 geht verloren bei w=3: Sender schickt N1, N2(verloren!), N3, N4, N5. Empfänger nimmt nur N1 an, wirft N3, N4, N5 weg. ACK bleibt bei ACK(1) hängen. Nach Timeout muss der Sender N2, N3, N4, N5 alle wiederholen – auch wenn N3-N5 beim ersten Mal ankamen!

Erweiterungen: NAK, Out-of-Order, kumulative ACKs

TechnikIdeeVorteil
Negative Quittung (NAK)Empfänger meldet Fehler sofort aktivKein Warten auf Timeout → schnellere Reaktion
Out-of-Order PufferEmpfänger speichert Pakete außer ReihenfolgeNur das wirklich fehlende Paket muss erneut gesendet werden
Kumulative ACKACK(N) bestätigt alles bis NWeniger ACK-Overhead
Selective ACK (SACK)Empfänger meldet einzelne BlöckeMinimale Wiederholungen (TCP-Erweiterung)
Merksatz

ACK(N) = kumulative Quittung bedeutet: „Ich habe N1, N2, … bis N alle korrekt erhalten." Erst das nächste erwartete Paket muss N+1 sein.

Nutzungseffizienz berechnen

Wie gut die Leitung ausgenutzt wird, hängt davon ab, wie viele Pakete der Sender gleichzeitig „in die Pipeline" stecken kann. Hier lernst du alle Formeln und das komplette Satelliten-Beispiel Schritt für Schritt.

Die vier wichtigen Zeiten

Formel
t_PDU = L / R

L = PDU-Größe in Bit · R = Übertragungsrate in bit/s → Wie lange braucht der Sender, um eine PDU auf die Leitung zu bringen?

Formel
t_ACK = a / R

a = ACK-Größe in Bit · R = Übertragungsrate → Wie lange braucht die Quittung auf dem Rückkanal? (Nicht vergessen – auch ACKs brauchen Zeit!)

Formel
T_cycle = t_PDU + 2 · t_sig + t_ACK

t_sig = einfache Signalverzögerung (Sender → Empfänger). Der Faktor 2 gilt für den vollen Umlauf (Hin + zurück). T_cycle = Zeit vom ersten Bit der PDU bis zum letzten Bit des ACK beim Sender.

Formel
η = (w · t_PDU) / T_cycle  [falls < 100%]

w = Fenstergröße. Wenn w · t_PDU ≥ T_cycle → Sender ist nie idle → η = 100 %. Sonst: Pipeline nicht voll, Leerlauf entsteht.

Formel
w_min = ⌈T_cycle / t_PDU⌉

Minimale ganzzahlige Fenstergröße für 100 % Auslastung (immer aufrunden!).

Vollständiges Beispiel: Satellitenkanal (Blatt 7)

R = 64 kbit/sÜbertragungsrate
L = 512 B = 4096 bitPDU-Größe
a = 8 B = 64 bitACK-Größe
t_sig = 270 msSignalverzögerung (einfach)
  1. Sendezeit pro PDU berechnen:
    t_PDU = L / R = 4096 bit / 64.000 bit/s = 0,064 s = 64 ms
  2. ACK-Sendezeit berechnen:
    t_ACK = a / R = 64 bit / 64.000 bit/s = 0,001 s = 1 ms
  3. Zykluszeit T_cycle berechnen:
    T_cycle = t_PDU + 2 · t_sig + t_ACK = 64 + 2 · 270 + 1 = 64 + 540 + 1 = 605 ms
  4. Pipeline-Check für jede Fenstergröße: w · t_PDU vs. T_cycle — wenn w · t_PDU ≥ 605 ms → 100 %
  5. Werte für w = 1, 7, 15 berechnen (siehe unten)
  6. Minimale Fenstergröße: w_min = ⌈605 / 64⌉ = ⌈9,453⌉ = 10

Schritt-für-Schritt für w = 7 (ausgerechnet)
η = (7 · 64 ms) / 605 ms = 448 / 605 ≈ 0,74 = 74 %

R_eff = η · R = 0,74 · 64 kbit/s ≈ 47,4 kbit/s
Alternativ direkt: R_eff = (w · L) / T_cycle = (7 · 4096 bit) / 0,605 s = 28.672 / 0,605 ≈ 47.400 bit/s

Merksatz: w_min

w_min = ⌈T_cycle / t_PDU⌉ = ⌈605 / 64⌉ = ⌈9,453⌉ = 10 — immer aufrunden! Mit w=9 wäre 9 · 64 = 576 < 605 → noch nicht 100 %. Mit w=10: 10 · 64 = 640 ≥ 605 ✓

Typische Fallen

❌ FalschT_cycle = t_PDU + 2 · t_sig (ACK-Zeit wird vergessen)
✅ RichtigT_cycle = t_PDU + 2 · t_sig + t_ACK. Der ACK wird ebenfalls über den Kanal übertragen und braucht Zeit. Bei a = 8 Byte, R = 64 kbit/s wären das immerhin 1 ms – in manchen Aufgaben kommt es auf jeden Millisekunde an!
❌ FalschRTT = t_sig = 270 ms (nur einfache Laufzeit)
✅ RichtigRTT = 2 · t_sig = 540 ms — das Signal muss von Sender zu Empfänger (270 ms) und der ACK wieder zurück (270 ms). t_sig ist immer die einfache Verzögerung.
❌ Falschw_min = T_cycle / t_PDU = 9,45 (keine Ganzzahl!)
✅ Richtigw muss eine ganze Zahl sein → aufrunden! w_min = ⌈9,45⌉ = 10. Mit w=9 wäre die Leitung noch nicht voll ausgelastet.
❌ FalschBei Go-Back-N quittiert der Empfänger N3, N4, N5, auch wenn N2 fehlt.
✅ RichtigOhne Out-of-Order-Puffer akzeptiert der Empfänger nur in Reihenfolge. N3, N4, N5 werden verworfen – der ACK bleibt bei ACK(1) stehen. Erst nach Timeout wird alles ab N2 wiederholt.
❌ FalschDas Sliding Window sendet erst alle w Pakete, wartet dann auf alle ACKs, sendet dann die nächsten w Pakete.
✅ RichtigDas Fenster gleitet: Sobald ACK(1) kommt, rutscht das Fenster und N4 kann sofort raus – nicht erst wenn alle anderen ACKs eingetroffen sind. Genau deshalb heißt es Sliding Window.
❌ FalschStop-and-Wait ist grundsätzlich schlechtes Design.
✅ RichtigStop-and-Wait ist bei niedriger Latenz oder geringem Durchsatzbedarf völlig OK (z.B. interaktive Steuerung kleiner Geräte). Schlecht wird es erst bei hoher Latenz (Satellit!) – dann wird η sehr klein.
