Worum geht's?

Wenn du eine Funktion mit einem Argument aufrufst, stellt sich die Frage: Wann wird das Argument berechnet? Sofort, bevor die Funktion startet? Oder erst, wenn die Funktion es wirklich braucht? Oder gar nie, falls es ungenutzt bleibt? Diese Entscheidung nennt sich Auswertungsstrategie — und Haskell trifft sie anders als Python oder Java.

Stell dir vor …

… du bestellst beim Bäcker ein Frühstück. Der Bäcker kann das so angehen:

  • Call-by-Value (CBV): Der Bäcker backt erst alle Brötchen frisch — egal ob du sie alle essen wirst — und bringt dann das fertige Tablett. (Python, Java, C++)
  • Call-by-Name (CBN): Der Bäcker bringt rohe Teiglinge. Du backst jedes Brötchen selbst, wenn du es essen willst — und falls du ein Brötchen zweimal isst, backst du es auch zweimal.
  • Call-by-Need / Lazy (Haskell): Wie CBN, aber der Bäcker merkt sich das Backergebnis. Jedes Brötchen wird nur einmal gebacken — und nur, wenn du es wirklich willst. Unendliche Tablettreihen sind kein Problem, solange du nur ein paar nimmst.

Auswertungsstrategien

Alle drei Strategien beschreiben dasselbe: Wie geht Haskell (oder eine andere Sprache) mit Argumenten um, bevor eine Funktion startet?

Call-by-Value (CBV)

In Sprachen wie Python, Java und C++ werden Argumente vor dem Funktionsaufruf komplett ausgewertet. Das ist vertraut und einfach vorherzusagen.

CBV-Eigenschaft

Jedes Argument muss terminieren. Wenn das Argument in einer Endlosschleife hängt, hängt auch der Aufruf — selbst wenn die Funktion es gar nicht benutzt.

-- CBV-Beispiel: f ([1,2] ++ [3,4])
-- mit f xs = head xs + length xs

-- Schritt 1: Argument auswerten:  [1,2] ++ [3,4]  =>  [1,2,3,4]
-- Schritt 2: f [1,2,3,4]
-- Schritt 3: head [1,2,3,4] + length [1,2,3,4]
-- Schritt 4: 1 + 4  =>  5

Call-by-Name (CBN)

Das Argument wird unausgewertet wie ein Textbaustein in die Funktion eingesetzt. Erst wenn die Funktion den Wert wirklich braucht, wird gerechnet. Nachteil: Falls das Argument mehrfach benutzt wird, rechnet man es auch mehrfach.

-- CBN-Beispiel: (\y -> if y<=1 then y*y else y) (1-1)

-- Schritt 1: (1-1) NICHT auswerten, direkt einsetzen:
--   if (1-1)<=1 then (1-1)*(1-1) else (1-1)
-- Schritt 2: Bedingung auswerten:  (1-1)<=1  =>  0<=1  =>  True
-- Schritt 3: then-Branch: (1-1)*(1-1)  =>  0*0  =>  0

Call-by-Need = Lazy Evaluation (Haskell)

Haskell verwendet Lazy Evaluation: wie CBN, aber mit einem entscheidenden Unterschied — jedes Argument wird nur einmal berechnet und das Ergebnis gecacht (gespeichert). Das vermeidet die Mehrfach-Berechnung von CBN.

CBVArgument sofort ausgewertet · muss terminieren
CBNArgument verzögert · evtl. mehrfach berechnet
Lazy (Haskell)CBN + Caching · jedes Argument max. 1× gerechnet
StrategieWann ausgewertet?Mehrfach-Berechnung?Sprachen
Call-by-ValueVor dem Aufruf—Python, Java, C++
Call-by-NameBei Bedarf (unkomprimiert)Ja, evtl.ALGOL 60
Call-by-Need (Lazy)Bei Bedarf + gecachtNeinHaskell

Unendliche Listen dank Lazy Evaluation

Weil Haskell nur das berechnet, was tatsächlich gebraucht wird, kannst du mit unendlichen Listen arbeiten — solange du nur einen endlichen Ausschnitt anforderst.

-- Unendliche Liste der natürlichen Zahlen
take 5 [1..]        -- => [1,2,3,4,5]

-- Unendliche Liste von Einsen
take 3 (repeat 1)   -- => [1,1,1]

-- Unendliche Fibonacci-Folge
fibs = 0 : 1 : zipWith (+) fibs (tail fibs)
take 5 fibs         -- => [0,1,1,2,3]

Terminationsbeweise

Wie beweist man, dass eine rekursive Funktion für alle erlaubten Eingaben irgendwann aufhört? Dafür gibt es drei Schritte:

  1. Abgeschlossenheit: Die Eingaben bleiben innerhalb der erlaubten Domain (z.B. n-1 ≥ 0 wenn n ≥ 1).
  2. Definiertheit: Jede Eingabe hat einen Weg zum Base Case — keine „Lücken".
  3. Abstiegsfunktion φ: Eine Funktion auf den Argumenten, die bei jedem rekursiven Aufruf streng kleiner wird und ein Minimum hat (z.B. φ(n) = n mit Minimum 0).
-- Beispiel: power x n (n >= 0)
power :: Integer -> Integer -> Integer
power _ 0 = 1
power x n = x * power x (n - 1)

-- φ(n) = n
-- power x n ruft power x (n-1) auf  →  φ(n) = n > n-1 = φ(n-1)  ✓
-- Minimum: φ(0) = 0  →  Base Case  ✓
-- Beispiel: odds xs (Liste von Int)
odds :: [Int] -> [Int]
odds []     = []
odds (x:xs)
  | x `mod` 2 == 0 = odds xs
  | otherwise       = x : odds xs

-- φ(xs) = length xs
-- odds (x:xs) ruft odds xs auf  →  length xs < length (x:xs)  ✓
-- Minimum: φ([]) = 0  →  Base Case  ✓
Merksatz: Lazy erlaubt Unendlichkeit

Nur weil Haskell lazy ist, kann take 5 [1..] funktionieren. Eine unendliche Liste wird nie vollständig erzeugt — nur so viele Elemente wie take anfordert. Mit CBV würde [1..] ewig laufen.

Schritt für Schritt: CBV-Auswertung von f ([1,2]++[3,4])

Rechnen wir das Klausur-Beispiel komplett durch — genau so, wie es in der Klausur erwartet wird (Call-by-Value):

f :: [Int] -> Int
f xs = head xs + length xs
  1. Ausgangspunkt: f ([1,2] ++ [3,4]) — das Argument ist noch unausgewertet.
  2. Argument zuerst auswerten (CBV!): [1,2] ++ [3,4] ergibt [1,2,3,4].
  3. Funktion aufrufen: f [1,2,3,4] — jetzt erst tritt die Funktion in Kraft.
  4. Body einsetzen: head [1,2,3,4] + length [1,2,3,4].
  5. Beide Seiten auswerten: head [1,2,3,4] = 1 und length [1,2,3,4] = 4.
  6. Ergebnis: 1 + 4 = 5.
f ([1,2] ++ [3,4]) → f [1,2,3,4] → head [1,2,3,4] + length [1,2,3,4]
→ 1 + 4 → 5

Zum Vergleich: CBN-Auswertung des Lambda-Ausdrucks

Jetzt schauen wir uns an, wie CBN mit demselben Prinzip umgeht — das Argument wird eingesetzt ohne ausgewertet zu werden:

-- Ziel: (\y -> if y <= 1 then y*y else y) (1-1)
-- mit Call-by-Name
  1. Argument NICHT auswerten: (1-1) bleibt wie es ist.
  2. Direkt einsetzen: y wird überall durch (1-1) ersetzt.
  3. Bedingung auswerten: (1-1) <= 1 → 0 <= 1 → True.
  4. Then-Branch wählen: (1-1) * (1-1).
  5. Ausmultiplizieren: 0 * 0 = 0.
(\y -> if y<=1 then y*y else y) (1-1) → if (1-1)<=1 then (1-1)*(1-1) else (1-1)
→ if True then (1-1)*(1-1) else (1-1) → (1-1)*(1-1) → 0*0 → 0
Merke: In der Klausur

Bei CBV: Als ersten Schritt immer das Argument auswerten, dann aufrufen. Bei CBN: Als ersten Schritt das Argument unausgewertet einsetzen (textuell ersetzen). Punkte gibt es oft genau für diesen ersten Schritt!

Typische Fallen (Klausur!)

❌ FalschBei CBV das Argument nicht zuerst auswerten: 
-- Falsche Reihenfolge:
f ([1,2] ++ [3,4])
= head ([1,2] ++ [3,4]) + length ([1,2] ++ [3,4])
  -- Argument wurde übersprungen!
✅ RichtigArgument ZUERST auswerten, dann erst die Funktion einsetzen: 
f ([1,2] ++ [3,4])
-- Schritt 1: [1,2] ++ [3,4]  =>  [1,2,3,4]
= f [1,2,3,4]
= head [1,2,3,4] + length [1,2,3,4]
= 1 + 4  =>  5
❌ FalschBei CBN das Argument trotzdem zuerst auswerten: 
-- Falsch: so wie CBV
(\y -> if y<=1 then y*y else y) (1-1)
-- Schritt 1: (1-1)  =>  0   <-- NEIN! Das ist CBV!
✅ RichtigBei CBN das Argument unausgewertet einsetzen: 
(\y -> if y<=1 then y*y else y) (1-1)
-- Schritt 1: y durch (1-1) ersetzen:
if (1-1)<=1 then (1-1)*(1-1) else (1-1)
❌ Falschfoldl statt foldl' bei großen Listen — Speicherleck durch aufgestaute Auswertung: 
-- Kann Speicher-Überlauf verursachen:
foldl (+) 0 [1..1000000]
✅ Richtigfoldl' aus Data.List erzwingt strenge Auswertung und braucht konstant Speicher: 
import Data.List (foldl')
foldl' (+) 0 [1..1000000]  -- O(1) Speicher
❌ FalschKein Minimum für die Abstiegsfunktion φ — dann ist kein Terminationsbeweis möglich: 
-- φ(n) = n ohne Bedingung n >= 0:
-- n kann negativ werden → kein Minimum → endlose Rekursion!
power x n = x * power x (n - 1)
-- ohne Base Case für n=0 oder n<0
✅ RichtigAbstiegsfunktion φ muss streng fallen UND ein Minimum besitzen: 
power _ 0 = 1              -- Base Case: φ(0)=0 ist Minimum
power x n = x * power x (n - 1)
-- φ(n) = n > n-1 = φ(n-1)  (streng absteigend)
-- φ(0) = 0  (Minimum erreicht  =>  terminiert)

🧩 Wissenscheck

Drei kurze Fragen — tippe deine Antwort ein oder klick die richtige Option. Grün = richtig.

1.Call-by-Value: Was ergibt f ([1,2] ++ [3,4]) mit f xs = head xs + length xs?

Zuerst das Argument: [1,2]++[3,4] = [1,2,3,4], dann head=1 + length=4 = 5.

2.Was wertet Haskell mit Lazy Evaluation (Call-by-Name) NICHT unbedingt aus?

Lazy heißt: ein Ausdruck wird nur berechnet, wenn sein Wert wirklich gebraucht wird — ungenutzte Argumente bleiben unausgewertet.

3.Wie viele Einsen liefert take 3 (repeat 1)? (repeat 1 ist unendlich — dank Lazy Evaluation kein Problem.)

take 3 zieht nur 3 Elemente aus der unendlichen Liste: [1,1,1] → 3 Einsen.
