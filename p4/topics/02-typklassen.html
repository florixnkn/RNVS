Worum geht's?

Polymorphie ist das Zauberwort: eine Definition, die für viele Typen passt. Statt für jeden Typ (Int, String, Bool) separate Funktionen zu schreiben, schreibst du generische Funktionen. Typklassen sind die Schnittstelle, die Verträge — sie sagen dem Compiler: „Dieser Typ verspricht, diese Methoden bereitzustellen."

Parametrische Polymorphie kennst du schon aus Rekursion:

data Box a = Box a              -- eine Box für jeden Typ
data Pair a b = Pair a b       -- ein Paar: zwei verschiedene Typen
data WTree a = Empty | Node (WTree a) a Int (WTree a)   -- ein gewichteter Baum

Der Parameter a (oder b) ist Typvariabel — sie wird erst beim Verwenden konkret. So kannst du Box Int, Box String, Box (Maybe Double) haben, alle mit demselben Code.

Stell dir vor …

… eine Schnittstellen-Beschreibung wie ein Steckdosen-Formular: Jeder Typ, der das Formular ausfüllt, bekommt automatisch alle Methoden. Functor = „ich kann meine Werte transformieren lassen" (wie eine Maschine, die in eine Kiste greift). Monad = „ich kann nacheinander Schritte ausführen, die alle meinen Kontext kennen" (wie ein Rezept, das jeden Schritt versteht, ob es regnet oder nicht).

Warum Typklassen? Weil Haskell sichere generische Programmierung will. Nicht „alle Typen können alles", sondern „dieser Typ verspricht diese Operationen".

Typklassen-Hierarchie

Die wichtigsten Typklassen bauen aufeinander auf:

Parametrische Polymorphie — der Anfang

Alle fünf Typklassen arbeiten mit generischen Datentypen:

-- Einfach: eine Schachtel für jeden Typ
data Box a = Box a

-- Zwei Typen: Pair a b = Pair a b

-- Rekursiv: ein gewichteter Baum
data WTree a = Empty | Node (WTree a) a Int (WTree a)
  -- Struktur: linker Unterbaum, Wert vom Typ a, Gewicht (Int), rechts

Der Punkt: Eine Definition, unendlich viele Typen (Box Int, Box String, WTree Bool, etc.).

Functor — Die Transformations-Schnittstelle

„Ich habe einen Wert in einem Kontext (einer Box, einer Liste, einem Baum). Lass mich die Werte transformieren, ohne den Kontext zu zerstören."

class Functor f where
  fmap :: (a -> b) -> f a -> f b
  -- fmap nimmt eine Funktion und eine Container-Struktur
  -- und wendet die Funktion auf alle inneren Werte an

Beispiele:

instance Functor Maybe where
  fmap f Nothing  = Nothing
  fmap f (Just x) = Just (f x)

instance Functor [] where
  fmap = map

instance Functor (Pair a) where
  fmap f (Pair x y) = Pair x (f y)  -- nur der letzte Parameter!

Gesetze (die jede Instanz erfüllen muss):

fmap id x == x — Identity (nichts zu tun = nichts ändert sich)

fmap (f . g) x == fmap f (fmap g x) — Komposition (Reihenfolge egal)

Foldable — Die Zusammenfassungs-Schnittstelle

„Ich habe viele Werte in einer Struktur. Lass mich sie zu einem Ergebnis zusammenfalten."

class Foldable t where
  fold   :: Monoid m => t m -> m
  foldMap :: Monoid m => (a -> m) -> t a -> m
  -- foldMap: Wende f auf jeden Wert an, kombiniere Ergebnisse mit <>

Beispiel:

instance Foldable [] where
  foldMap f []     = mempty
  foldMap f (x:xs) = f x <> foldMap f xs

instance Foldable Maybe where
  foldMap f Nothing  = mempty
  foldMap f (Just x) = f x

Applicative — Effekte kombinieren

„Ich habe zwei Boxen: eine mit einer Funktion, eine mit einem Wert. Ich will sie kombinieren."

class Functor f => Applicative f where
  pure  :: a -> f a          -- Wert in die Box packen
  (<*>) :: f (a -> b) -> f a -> f b  -- Funktion-Box <*> Wert-Box

Beispiel:

instance Applicative Maybe where
  pure x = Just x
  Nothing   <*> _ = Nothing
  _         <*> Nothing = Nothing
  (Just f)  <*> (Just x) = Just (f x)

Monad — Die Kontextstepperei

„Ich führe Schritte aus, die jeweils den Kontext kennen. Jeder Schritt kann vom Ergebnis des vorherigen abhängen."

class Applicative m => Monad m where
  return :: a -> m a
  (>>=)  :: m a -> (a -> m b) -> m b
  -- >>= spricht: „Gib mir ein m a, eine Funktion (a -> m b),
  --              und ich kombiniere alles zu m b"

Beispiel:

instance Monad Maybe where
  return = Just
  Nothing  >>= f = Nothing
  (Just x) >>= f = f x
  -- Wenn x existiert, wende f an. Sonst: Nothing bleibt Nothing.

Ord — Die Ordnungs-Schnittstelle

„Meine Werte sind vergleichbar: kleiner, gleich, größer."

class Eq a => Ord a where
  compare :: a -> a -> Ordering  -- LT | EQ | GT
  (<=) :: a -> a -> Bool
  -- und weitere Operatoren
Typklasse Signatur (gekürzt) Intuition
Functor f fmap :: (a -> b) -> f a -> f b Werte in einem Kontext transformieren
Foldable t foldMap :: Monoid m => (a -> m) -> t a -> m Alle Werte zusammenfalten
Applicative f (<*>) :: f (a -> b) -> f a -> f b Funktions- und Wert-Box kombinieren
Monad m (>>=) :: m a -> (a -> m b) -> m b Kontextabhängige Schritte sequenzen
Ord a compare :: a -> a -> Ordering Werte vergleichen und sortieren
Polymorphe Datentypen sind nicht automatisch Functor. Du musst eine Instanz schreiben und die Gesetze beweisen.
fmap transformiert immer nur den letzten Typ-Parameter. Bei Pair a b wird nur b transformiert, nicht a.
foldMap braucht ein Monoid: Ein Typ mit mempty (neutral) und <> (kombinieren).
Monad = Applicative + Abhängigkeiten. Mit >>= kann ein Schritt vom vorherigen Ergebnis abhängen.

Instanzen schreiben

Jetzt ein echtes Beispiel: Wir schreiben Instanzen für unseren gewichteten Baum.

WTree: Foldable mit Gewicht-Filter

Der Baum hat Empty (Blatt) und Node (innerer Knoten mit Gewicht). Wir wollen nur Knoten mit Gewicht > 0 falten:

data WTree a = Empty | Node (WTree a) a Int (WTree a)

instance Foldable WTree where
  foldMap f Empty = mempty
  foldMap f (Node l x w r)
    | w > 0    = foldMap f l <> f x <> foldMap f r
    | otherwise = mempty  -- Knoten mit w <= 0 überspringen

Was passiert hier?

  • Empty → mempty (neutrales Element)
  • Node mit w > 0 → falte rekursiv linken und rechten Unterbaum, kombiniere mit f x mittels <>
  • Node mit w <= 0 → überspringe den Knoten, liefere mempty

Traversal-Ordnungen: Preorder, Inorder, Postorder

Die Ordnung, in der wir den Baum durchlaufen, ist entscheidend:

-- Preorder: Knoten, dann links, dann rechts
preorder :: WTree a -> [a]
preorder Empty = []
preorder (Node l x w r) = [x] ++ preorder l ++ preorder r

-- Inorder: Links, Knoten, rechts
inorder :: WTree a -> [a]
inorder Empty = []
inorder (Node l x w r) = inorder l ++ [x] ++ inorder r

-- Postorder: Links, rechts, dann Knoten
postorder :: WTree a -> [a]
postorder Empty = []
postorder (Node l x w r) = postorder l ++ postorder r ++ [x]

Beispiel: Baum mit Knoten A(links→B, rechts→C):

  • Preorder: A, B, C (Vater zuerst)
  • Inorder: B, A, C (Vater in der Mitte)
  • Postorder: B, C, A (Vater zuletzt)

Zipper: Functor & Foldable

Ein Zipper ist ein Baum mit „Fokus" — ein Knoten ist markiert:

data Zipper a = Zipper (WTree a) (WTree a)  -- (Fokus, Rest)

instance Functor Zipper where
  fmap f (Zipper focus rest) = Zipper (fmap f focus) (fmap f rest)

instance Foldable Zipper where
  foldMap f (Zipper focus rest) = foldMap f focus <> foldMap f rest

Peano: Ord-Instanz

Natürliche Zahlen nach Peano können wir ordnen:

data Peano = Zero | Succ Peano

instance Eq Peano where
  Zero == Zero = True
  (Succ a) == (Succ b) = a == b
  _ == _ = False

instance Ord Peano where
  compare Zero Zero = EQ
  compare Zero (Succ _) = LT
  compare (Succ _) Zero = GT
  compare (Succ a) (Succ b) = compare a b
Instanz-Checkliste
  • Alle Methoden der Typklasse implementiert?
  • Alle Gesetze erfüllt? (z.B. Functor-Gesetze für fmap)
  • Rekursion auf alle Unterbäume angewandt?
  • Spezialfälle behandelt? (Empty, Nothing, Zero, etc.)
  • Hierarchie beachtet? (z.B. Functor vor Applicative vor Monad)

Schritt für Schritt

Beispiel 1: fmap (+1) (Just 5)

Wie wertet Haskell diesen Ausdruck aus? Wir folgen der Maybe-Instanz von Functor:

  1. Ausdruck: fmap (+1) (Just 5) — Funktor-Instanz für Maybe wird gesucht.
  2. Passendes Muster: fmap f (Just x) = Just (f x) — hier ist f = (+1), x = 5.
  3. Substitution: Just ((+1) 5) — die Funktion wird auf den Wert im Kontext angewandt.
  4. Arithmetik: Just 6 — fertig! Die Kiste bleibt zu, ihr Inhalt wurde transformiert.
fmap (+1) (Just 5) → Just ((+1) 5) → Just 6
fmap (+1) Nothing → Nothing (kein Wert, nichts zu tun)

Beispiel 2: foldMap Sum [1,2,3]

Eine Liste wird mit dem Monoid Sum gefaltet. Sum hat mempty = Sum 0 und Sum a <> Sum b = Sum (a+b):

  1. Start: foldMap Sum [1,2,3] — wandle jeden Wert mit Sum um, kombiniere.
  2. Aufrollen: Sum 1 <> foldMap Sum [2,3] — erstes Element wird direkt verarbeitet.
  3. Rekursion: Sum 1 <> (Sum 2 <> (Sum 3 <> mempty)) — alle Elemente ausgerollt.
  4. Kombination: Sum 1 <> Sum 2 = Sum 3, dann Sum 3 <> Sum 3 = Sum 6.
  5. Ergebnis: Sum 6 — die Summe aller Elemente.
foldMap Sum [1,2,3] → Sum 1 <> Sum 2 <> Sum 3 <> mempty → Sum 6

Beispiel 3: foldMap mit Gewichts-Filter (WTree)

Baum: Node Empty 'A' 3 Empty — einzelner Knoten, Gewicht 3:

foldMap f (Node Empty 'A' 3 Empty) → (w=3 > 0, also falten)
→ foldMap f Empty <> f 'A' <> foldMap f Empty → mempty <> f 'A' <> mempty → f 'A'
Merksatz

fmap = Kiste bleibt zu, Inhalt wird transformiert. foldMap = alle Elemente werden zu einem Monoid-Wert kombiniert. Beide nutzen rekursive Instanzen.

Typische Fallen (Klausur!)

❌ FalschFunctor-Instanz für Pair a — beide Felder transformieren: 
instance Functor (Pair a) where
  fmap f (Pair x y) = Pair (f x) (f y)  -- BEIDE!
✅ RichtigFunctor transformiert immer nur den letzten Typ-Parameter. Bei Pair a b ist das b: 
instance Functor (Pair a) where
  fmap f (Pair x y) = Pair x (f y)  -- nur y!
❌ FalschFoldable: [] statt mempty für leere Struktur: 
instance Foldable WTree where
  foldMap f Empty = []  -- Falscher Typ!
✅ Richtigmempty ist das neutrale Element des jeweiligen Monoids (Sum 0, "", []…): 
instance Foldable WTree where
  foldMap f Empty = mempty
❌ FalschApplicative ohne Functor-Instanz — der Compiler verweigert das: 
-- Vergessen: instance Functor (Pair a) where ...

instance Applicative (Pair a) where
  pure x = Pair undefined x
  (Pair f g) <*> (Pair x y) = Pair x (g y)
✅ RichtigFunctor-Instanz zuerst schreiben, dann Applicative aufbauen: 
instance Functor (Pair a) where
  fmap f (Pair x y) = Pair x (f y)

instance Applicative (Pair a) where
  pure x = Pair undefined x
  (Pair f g) <*> (Pair x y) = Pair x (g y)
❌ FalschOrd mit zirkulärer Implementation — compare ist nirgends definiert: 
instance Ord Peano where
  a <= b = compare a b == LT  -- compare existiert nicht!
✅ Richtigcompare direkt definieren — alle anderen Operatoren folgen daraus: 
instance Ord Peano where
  compare Zero Zero     = EQ
  compare Zero (Succ _) = LT
  compare (Succ _) Zero = GT
  compare (Succ a) (Succ b) = compare a b
❌ FalschFoldable WTree: Gewicht w ignorieren — alle Knoten werden gefaltet: 
instance Foldable WTree where
  foldMap f Empty = mempty
  foldMap f (Node l x w r)
    = foldMap f l <> f x <> foldMap f r  -- w ignoriert!
✅ RichtigGuard | w > 0 prüft das Gewicht — nur positive Knoten werden berücksichtigt: 
instance Foldable WTree where
  foldMap f Empty = mempty
  foldMap f (Node l x w r)
    | w > 0     = foldMap f l <> f x <> foldMap f r
    | otherwise = foldMap f l <> foldMap f r

🎯 Quick-Reference für die Klausur

Typklasse Haupt-Methode(n) Minimal-Implementation
Functor fmap 
instance Functor Box where
  fmap f (Box x) = Box (f x)
Foldable foldMap 
instance Foldable Box where
  foldMap f (Box x) = f x
Applicative pure, (<*>) 
instance Applicative Box where
  pure x = Box x
  (Box f) <*> (Box x) = Box (f x)
Monad return, (>>=) 
instance Monad Box where
  return x = Box x
  (Box x) >>= f = f x
Ord compare 
instance Ord Int where
  compare a b | a < b = LT
              | a > b = GT
              | otherwise = EQ

Checkliste vor der Klausur

  • ✓ Hierarchie merken: Functor < Applicative < Monad; Eq < Ord; Foldable unabhängig
  • ✓ fmap: transformiert nur letzten Parameter
  • ✓ foldMap: braucht Monoid, mempty für leere Struktur
  • ✓ <*>: kombiniert Funktion-Box mit Wert-Box
  • ✓ >>=: rechte Seite hängt vom linken Ergebnis ab
  • ✓ compare: ist die Basis für alle Ordnungs-Operatoren
  • ✓ Gewichte prüfen: Guards (|) nutzen!
  • ✓ HTML-Escape: <, >, &, => in Code-Blöcken

🧩 Wissenscheck

Drei kurze Fragen — tippe deine Antwort ein oder klick die richtige Option. Grün = richtig.

1.Bei instance Functor (Pair a) mit data Pair a b = Pair a b — welcher Wert wird von fmap transformiert?

Functor transformiert immer nur den letzten Typ-Parameter: fmap f (Pair x y) = Pair x (f y).

2.Was ergibt fmap (+1) (Just 5)?

fmap wendet die Funktion auf den Wert im Kontext an: Just (5+1) = Just 6.

3.Was liefert foldMap f Empty für eine leere Struktur?

mempty ist das neutrale Element des jeweiligen Monoids — nicht fest [] oder 0.
