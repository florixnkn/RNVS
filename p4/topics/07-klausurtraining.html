Aufgabe 1 (9 Punkte): Polynom-Auswertung per Horner-Schema

Original-Aufgabentext: Ein Polynom P(x) = a₀ + a₁·x + … + aₙ·xⁿ soll aus der Koeffizientenliste [a0,...,an] berechnet werden — mithilfe der Horner-Umformung:

Horner-Schema
P(x) = a₀ + x(a₁ + x(a₂ + … + x(aₙ₋₁ + x·aₙ)…))

Statt jede Potenz xⁱ einzeln zu berechnen, wird von innen nach außen ausgeklammert — nur noch Additionen und Multiplikationen, keine einzige Potenz muss explizit berechnet werden.

(a) 3 P. — poly :: [Double] -> Double -> Double

Original-Aufgabentext: Definiere poly, sodass poly as x den Wert des durch as beschriebenen Polynoms an der Stelle x liefert.

poly :: [Double] -> Double -> Double
poly [] _ = 0
poly (a:as) x = a + x * poly as x

Das ist die direkte, nicht-endrekursive Übersetzung des Horner-Schemas: Der erste Koeffizient wird addiert, der Rest der Liste rekursiv ausgewertet und mit x multipliziert — genau die Klammerstruktur von oben, nur von links nach rechts aufgebaut.

(b) 4 P. — endrekursive Variante poly'

Original-Aufgabentext: Vervollständige das Gerüst zu einer endrekursiven Variante:

poly' :: [Double] -> Double -> Double
poly' cs x = aux (reverse cs) 0
  where
    aux [] acc = ...
    aux (a:as) acc = ...

Musterlösung:

poly' :: [Double] -> Double -> Double
poly' cs x = aux (reverse cs) 0
  where
    aux [] acc = acc
    aux (a:as) acc = aux as (acc * x + a)

reverse cs liefert [an,...,a0] — die Koeffizienten in umgekehrter Reihenfolge. Beim Abarbeiten wird der Akkumulator acc jeweils mit x multipliziert und der nächste (kleinere) Koeffizient addiert — klassisches Horner-Schema als Akkumulator-Rekursion, bei jedem Schritt bereits vollständig ausgewertet (kein aufgeschobener Rechenbaum wie bei poly).

(c) 2 P. — poly'' ohne explizite Rekursion

Original-Aufgabentext: Definiere poly'' ohne explizite Rekursion, mithilfe von foldr :: (a -> b -> b) -> b -> t a -> b.

poly'' :: [Double] -> Double -> Double
poly'' as x = foldr (\a acc -> a + x * acc) 0 as

foldr faltet die Liste von rechts nach links — für jeden Koeffizienten a und den bereits gefalteten Rest acc wird genau die Horner-Formel a + x * acc angewendet. Das ist strukturell identisch zu poly aus (a), nur ohne die Rekursion selbst hinzuschreiben.

❌ FalschBei (b) das reverse vergessen oder die Reihenfolge der Koeffizienten verwechseln: 
poly' cs x = aux cs 0   -- FALSCH: ohne reverse
  where aux (a:as) acc = aux as (acc * x + a)
-- verarbeitet a0 zuerst statt an — falsche Klammerung!
✅ Richtigreverse cs nicht vergessen, damit an als Erstes in den Akkumulator einfließt: 
poly' cs x = aux (reverse cs) 0
  where aux (a:as) acc = aux as (acc * x + a)
❌ FalschBei (c) foldl statt foldr verwenden — passt nicht zur Horner-Klammerung: 
poly'' as x = foldl (\acc a -> a + x * acc) 0 as  -- FALSCH herum
✅ Richtigfoldr passt natürlich zur Horner-Rekursion — sie baut den Ausdruck von rechts (dem höchsten Grad) her auf: 
poly'' as x = foldr (\a acc -> a + x * acc) 0 as

Aufgabe 2 (6 Punkte): Auswertungsstrategien erkennen

Original-Aufgabentext: Gegeben sei f x y = x * x + y und der Ausdruck f (3 + 4) (2 - 1). Es folgen drei Reduktionsfolgen — benenne jeweils die verwendete Auswertungsstrategie.

(a) 2 P.

f (3+4) (2-1) → f 7 (2-1) → f 7 1
→ 7*7+1 → 49+1 → 50

Musterlösung: Call-by-Value / strikte Auswertung (applicative order). Beide Argumente werden vor dem Funktionsaufruf jeweils vollständig zu einem Wert reduziert (erst (3+4)→7, dann (2-1)→1), erst dann wird in den Funktionskörper eingesetzt.

(b) 2 P.

f (3+4) (2-1) → (3+4)*(3+4)+(2-1) → 7*7+(2-1)
→ 7*7+1 → 49+1 → 50

Musterlösung: Call-by-Need / lazy mit Sharing (Haskells Standardstrategie). Die unausgewerteten Ausdrücke werden zuerst eingesetzt (x zweimal als (3+4)), aber beide Vorkommen von (3+4) werden gemeinsam in einem Schritt zu 7 reduziert — das zeigt, dass sich beide Vorkommen einen (gesharten) Thunk teilen: wird einer ausgewertet, sind automatisch beide fertig.

(c) 2 P.

f (3+4) (2-1) → (3+4)*(3+4)+(2-1) → 7*(3+4)+(2-1)
→ 7*7+(2-1) → 7*7+1 → 49+1 → 50

Musterlösung: Call-by-Name / lazy ohne Sharing. Auch hier wird zuerst unausgewertet eingesetzt, aber die beiden Vorkommen von (3+4) werden unabhängig voneinander nacheinander reduziert (erst das linke, dann separat das rechte) — kein Sharing, doppelte Arbeit.

❌ Falsch(b) und (c) für identisch halten, weil beide Folgen fast gleich aussehen (beide substituieren erst, werten spät aus): 
-- Beide Folgen enthalten "(3+4)*(3+4)+(2-1)" als Zwischenschritt
-- → sieht auf den ersten Blick nach derselben Strategie aus!
✅ RichtigGenau hinschauen, ob in EINEM Schritt beide Vorkommen auf einmal reduziert werden (Sharing = Call-by-Need) oder ob es zwei getrennte Schritte sind (kein Sharing = Call-by-Name): 
-- (b): (3+4)*(3+4)+... -> 7*7+...      -- EIN Schritt fuer beide Vorkommen
-- (c): (3+4)*(3+4)+... -> 7*(3+4)+...  -- ZWEI getrennte Schritte

Aufgabe 3 (24 Punkte): Entscheidungsbäume — ADT, Foldable, Eq

Original-Aufgabentext:

data DT k v = Empty | Leaf v | Node (k -> Bool) (DT k v) (DT k v)

Beispiel: dt ordnet jeder Zahl außer 0 einen String zu (True im Prädikat → linker Teilbaum):

dt = Node (==0)
       Empty
       (Node (>0)
          (Node odd (Leaf "Ungerade") (Leaf "Gerade"))
          (Leaf "Negativ"))

(a) 5 P. — find :: DT k v -> k -> Maybe v

Original-Aufgabentext: Definiere find, das einen Baum nach dem Wert für einen gegebenen Schlüssel durchsucht.

find :: DT k v -> k -> Maybe v
find Empty _ = Nothing
find (Leaf v) _ = Just v
find (Node p l r) k = if p k then find l k else find r k

(b) 1 P. — Kind von DT

Original-Aufgabentext: Welchen Kind hat DT?

DT :: * -> * -> *

DT hat zwei Typparameter (k und v) und liefert erst nach Angabe beider einen konkreten Typ — deshalb zwei Pfeile im Kind.

(c) 5 P. — Foldable-Instanz

Original-Aufgabentext: Definiere eine Foldable-Instanz für DT k über foldMap :: Monoid m => (a -> m) -> t a -> m.

instance Foldable (DT k) where
  foldMap _ Empty = mempty
  foldMap f (Leaf v) = f v
  foldMap f (Node _ l r) = foldMap f l <> foldMap f r

Bei Empty gibt es nichts beizutragen → mempty. Bei Leaf v wird f auf den einen Wert angewendet. Bei Node müssen die Ergebnisse beider Teilbäume mit <> (Monoid-Verknüpfung) zusammengeführt werden — das Prädikat selbst spielt für foldMap keine Rolle.

(d) 2 P. — valueList, pointfree

Original-Aufgabentext: Definiere valueList :: DT k v -> [v] pointfree, nur mit foldMap und : sowie [].

valueList :: DT k v -> [v]
valueList = foldMap (:[])

(:[]) verpackt jeden einzelnen Wert in eine Einzel-Liste — der Monoid für Listen ist (++), also klebt foldMap automatisch alle Einzel-Listen zu einer Gesamtliste zusammen.

(e) 5 P. — Eq-Instanz

Original-Aufgabentext: Zwei Bäume sind gleich, wenn sie die gleichen Werte in der gleichen Struktur enthalten (Prädikate werden dabei ignoriert) — definiere das ohne die Funktionen aus (a)–(d), mit geeigneten Typklasseneinschränkungen.

instance Eq v => Eq (DT k v) where
  Empty == Empty = True
  (Leaf v1) == (Leaf v2) = v1 == v2
  (Node _ l1 r1) == (Node _ l2 r2) = l1 == l2 && r1 == r2
  _ == _ = False

Die Einschränkung Eq v => wird gebraucht, weil in der Leaf-Gleichung Werte vom Typ v mit == verglichen werden. Für k ist keine Einschränkung nötig — die Prädikat-Funktionen werden im Node-Fall mit _ ignoriert.

(f) 6 P. — simplify

Original-Aufgabentext: Definiere simplify :: Eq v => DT k v -> DT k v, das einen Baum rekursiv vereinfacht: Sind die beiden Teilbäume eines Node nach der Eq-Instanz gleich, werden sie zum linken zusammengefasst. Ist ein Teilbaum Empty, wird er durch den nicht-leeren ersetzt.

simplify :: Eq v => DT k v -> DT k v
simplify Empty = Empty
simplify (Leaf v) = Leaf v
simplify (Node p l r) =
  let l' = simplify l
      r' = simplify r
  in if l' == r' then l'
     else case (l', r') of
       (Empty, _)    -> r'
       (_, Empty)    -> l'
       _             -> Node p l' r'

Erst werden beide Teilbäume rekursiv vereinfacht (l', r') — bottom-up. Erst danach wird geprüft: Sind die vereinfachten Teilbäume gleich, reicht einer davon. Ist genau einer Empty, bleibt nur der andere übrig. Sonst bleibt der Node mit den (bereits vereinfachten) Teilbäumen bestehen.

❌ FalschBei (c) im Node-Fall nur einen Teilbaum falten und den anderen vergessen: 
foldMap f (Node _ l r) = foldMap f l   -- FALSCH: r fehlt!
✅ RichtigBeide Teilbäume mit <> kombinieren: 
foldMap f (Node _ l r) = foldMap f l <> foldMap f r
❌ FalschBei (e) versehentlich die Prädikat-Funktionen mit vergleichen: 
(Node p1 l1 r1) == (Node p2 l2 r2) = p1 == p2 && l1 == l2 && r1 == r2
-- FALSCH: Funktionen sind i.A. NICHT vergleichbar (keine Eq-Instanz)!
✅ RichtigPrädikate mit _ ignorieren — laut Aufgabe zählen nur Werte und Struktur: 
(Node _ l1 r1) == (Node _ l2 r2) = l1 == l2 && r1 == r2
❌ FalschBei (f) zuerst auf Empty/Gleichheit prüfen und erst danach (oder gar nicht) rekursiv vereinfachen: 
simplify (Node p l r)
  | l == r = l              -- FALSCH: l und r wurden noch NICHT vereinfacht!
  | otherwise = Node p (simplify l) (simplify r)
✅ RichtigErst rekursiv vereinfachen (bottom-up), DANN erst auf Empty/Gleichheit prüfen: 
simplify (Node p l r) =
  let l' = simplify l
      r' = simplify r
  in if l' == r' then l' else ...

Aufgabe 4 (10 Punkte): Terminationsbeweis für skip

skip :: Integer -> [a] -> [a]
skip 0 xs = xs
skip _ [] = []
skip n (_:xs) = skip (n - 1) xs

Original-Aufgabentext: Quellbereich ist A = ℤ × a* (Paare aus ganzen Zahlen und Listen). Zu zeigen ist Termination auf der Definitionsmenge A' = ℕ₀ × a<ω (nicht-negative Zahlen, endliche Listen).

(a) 1 P. — Wert von skip 3 [1..5]

skip 3 [1,2,3,4,5]  =  [4,5]

Drei Elemente werden übersprungen (Pattern 3 feuert dreimal), übrig bleiben [4,5].

(b) 2 P. — Beispiel eines nicht-terminierenden Aufrufs

skip (-1) [1..]        -- oder: skip (-1) (repeat 0)

n wird nie 0 (es wird bei jedem Schritt nur negativer), und die Liste geht nie aus — also feuert weder Pattern 1 (n=0) noch Pattern 2 (xs=[]) jemals, die Rekursion läuft endlos.

(c) 3 P. — Abgeschlossenheit

Original-Aufgabentext: Zeige: Für (n,xs) ∈ A' liegt jeder rekursive Aufruf wieder in A'.

Musterlösung: Nur der dritte Fall (n≠0, xs=(_:xs')) rekursiert, zu skip (n-1) xs'. Da Pattern 1 (n=0) nicht griff, ist n≥1 (n∈ℕ₀), also n-1≥0, also n-1∈ℕ₀. xs' ist der Rest einer endlichen Liste, also wieder endlich. Also (n-1,xs')∈A'.

(d) 2 P. — Definiertheit

Original-Aufgabentext: Zeige: Jedes (n,xs)∈A' wird von genau einem Pattern erfasst.

Musterlösung: Ist n=0, matcht Pattern 1 (unabhängig von xs). Ist n≠0 und xs=[], matcht Pattern 2. Ist n≠0 und xs=(_:xs'), matcht Pattern 3. Da xs stets entweder [] oder (_:xs') ist (endliche Liste) und n entweder 0 oder ≠0, ist stets genau ein Pattern (in Bearbeitungsreihenfolge das erste passende) einschlägig — die Funktion ist auf ganz A' definiert.

(e) 2 P. — Abstiegsfunktion

Original-Aufgabentext: Gib eine Abstiegsfunktion m: A' → ℕ₀ an.

m (n, xs) = length xs

Bei jedem rekursiven Aufruf skip n (_:xs) = skip (n-1) xs gilt m(n,(_:xs)) = length xs + 1 > length xs = m(n-1,xs), die Funktion sinkt also strikt und ist stets ≥0 — damit terminiert jede Rekursionskette (die Liste wird endlich oft kürzer, bis Pattern 1 oder 2 greift).

❌ FalschNaheliegend, aber falsch: m(n,xs) = n als Abstiegsfunktion nehmen: 
m (n, xs) = n   -- FUNKTIONIERT NICHT!
✅ Richtign kann negativ anfangen oder gar nicht bis 0 sinken, bevor die Liste leer ist — length xs ist der robustere Maßstab, weil Pattern 3 die Liste immer um genau 1 verkürzt: 
m (n, xs) = length xs   -- sinkt garantiert bei jedem rekursiven Aufruf

Aufgabe 5 (11 Punkte): Monadische Wiederholung — repeatUntil

game :: IO ()
game = do
  repeatUntil (== 42) askNumber
  putStrLn "Richtig geraten!"

(a) 5 P. — repeatUntil

Original-Aufgabentext: Definiere repeatUntil :: Monad m => (a -> Bool) -> m a -> m (), das die übergebene Aktion so lange ausführt, bis der gelieferte Wert das Prädikat erfüllt.

repeatUntil :: Monad m => (a -> Bool) -> m a -> m ()
repeatUntil p action = do
  x <- action
  if p x then return () else repeatUntil p action

Jeder Durchlauf führt action aus und packt das Ergebnis mit <- aus. Erfüllt x das Prädikat, ist repeatUntil fertig (return ()). Sonst ruft sich repeatUntil mit derselben Aktion erneut auf — dadurch wird action ein weiteres Mal ausgeführt und kann (z. B. bei IO) einen neuen Wert liefern.

(b) 4 P. — askNumber

Original-Aufgabentext: Definiere askNumber :: IO Int, das zur Eingabe auffordert, mit getLine liest, in eine Integer umwandelt und zurückgibt. Nutze getLine :: IO String und putStr :: String -> IO ().

askNumber :: IO Int
askNumber = do
  putStr "Rate eine Zahl! "
  line <- getLine
  return (read line)

getLine liest die Nutzereingabe als String ein und packt ihn mit <- in line aus. read line wandelt den String in eine Zahl um, return hebt das Ergebnis zurück in den IO-Kontext.

(c) 2 P. — Warum nicht rein funktional?

Original-Aufgabentext: Warum lässt sich repeatUntil NICHT sinnvoll durch eine rein funktionale Variante pureRepeatUntil :: (a -> Bool) -> a -> () ersetzen?

Musterlösung: Der Witz von repeatUntil ist, dieselbe Aktion mehrfach auszuführen, um bei jedem Durchlauf einen neuen (potenziell unterschiedlichen) Wert zu erhalten — z. B. eine neue Nutzereingabe. Eine reine Funktion liefert für denselben Eingabewert immer dasselbe Ergebnis; ein fester Wert a lässt sich nicht „wiederholt neu einlesen". Ohne einen monadischen Kontext (IO) gibt es keine Möglichkeit, bei jedem Rekursionsschritt eine frische, potenziell andere Eingabe zu bekommen — die Funktion würde entweder sofort terminieren (wenn p a schon gilt) oder für immer dieselbe (falsche) Prüfung wiederholen (Endlosschleife ohne Fortschritt).

❌ FalschBei (a) vergessen, dass sich repeatUntil rekursiv mit derselben Aktion erneut aufruft — stattdessen versuchen, mit dem bereits ausgepackten Wert weiterzumachen: 
repeatUntil p action = do
  x <- action
  if p x then return () else repeatUntil p (return x)  -- FALSCH: liest nie neu!
✅ RichtigImmer mit der ursprünglichen Aktion erneut aufrufen, damit bei jedem Durchlauf frisch ausgeführt wird: 
repeatUntil p action = do
  x <- action
  if p x then return () else repeatUntil p action
Gut zu wissen

Bei (b) kann read line bei einer Fehleingabe (z. B. Buchstaben statt Zahl) zur Laufzeit crashen — das ist hier okay, weil die Klausuraufgabe keine Fehlerbehandlung verlangt.

Merksatz für die Klausur

Horner: foldr, nicht foldl. CBV wertet zuerst aus, CBN/Need setzen erst ein — der Unterschied zwischen CBN und CBN-mit-Need ist einzig das Sharing bei mehrfachen Vorkommen. Bei ADT-Eq-Instanzen niemals Funktionen (Prädikate) vergleichen. Bei Terminationsbeweisen: Abstiegsfunktion an dem messen, was garantiert bei jedem Aufruf kleiner wird (hier: die Listenlänge, nicht der Zähler). Monadische Wiederholung ruft dieselbe Aktion erneut auf — nur so gibt es bei jedem Durchlauf einen frischen Wert.

1.🎓 Nachklausur 25.09.2025: Aufgabe 1a — Gegeben poly [] _ = 0 und poly (a:as) x = a + x * poly as x. Was liefert poly [1,2,3] 2?

Horner von rechts einsetzen: poly [1,2,3] 2 = 1 + 2·(2 + 2·(3 + 2·poly [] 2)) = 1 + 2·(2 + 2·3) = 1 + 2·8 = 17 — passt zu P(x) = 1 + 2x + 3x² bei x=2: 1+4+12=17.

2.🎓 Nachklausur 25.09.2025: Aufgabe 1c — Welche Standard-Fold-Funktion (foldr oder foldl) passt natürlich zur Horner-Rekursion „von rechts" und wird für poly'' verwendet?

poly'' as x = foldr (\a acc -> a + x * acc) 0 as — foldr faltet von rechts nach links durch die Liste, genau wie die Horner-Rekursion selbst. foldl würde stattdessen von links falten und die Reihenfolge vertauschen.

3.🎓 Nachklausur 25.09.2025: Aufgabe 2a — Welche Auswertungsstrategie zeigt diese Reduktionsfolge? f (3+4) (2-1) → f 7 (2-1) → f 7 1 → 7*7+1 → 49+1 → 50

Beide Argumente werden vor dem Aufruf vollständig zu einem Wert reduziert (erst (3+4)→7, dann (2-1)→1) — das ist Call-by-Value.

4.🎓 Nachklausur 25.09.2025: Aufgabe 2b — In dieser Folge werden BEIDE Vorkommen von (3+4) in einem gemeinsamen Schritt zu 7: f (3+4) (2-1) → (3+4)*(3+4)+(2-1) → 7*7+(2-1) → 7*7+1 → 49+1 → 50. Welche Strategie ist das?

Die zwei Vorkommen von (3+4) werden GEMEINSAM in einem Schritt zu 7 reduziert — das zeigt einen gesharten Thunk. Genau das ist Call-by-Need, Haskells Standardstrategie.

5.🎓 Nachklausur 25.09.2025: Aufgabe 3a — Für dt = Node (==0) Empty (...) und find :: DT k v -> k -> Maybe v: Was liefert find dt 0?

Für k=0 gilt (==0) 0 = True → find geht in den linken Teilbaum Empty → find Empty _ = Nothing.

6.🎓 Nachklausur 25.09.2025: Aufgabe 3b — Welchen Kind hat DT (zwei Typparameter k und v)?

DT braucht zwei Typparameter, bevor ein konkreter Typ entsteht — deshalb zwei Pfeile: * -> * -> *.

7.🎓 Nachklausur 25.09.2025: Aufgabe 4a — Was liefert skip 3 [1..5]?

Drei Elemente werden übersprungen (Pattern 3 feuert dreimal), übrig bleibt [4,5].

8.🎓 Nachklausur 25.09.2025: Aufgabe 4e — Welche Abstiegsfunktion m: A' → ℕ₀ beweist Termination von skip?

Die naheliegende Wahl m(n,xs)=n funktioniert nicht, weil n negativ anfangen oder gar nicht bis 0 sinken muss, bevor die Liste leer ist. m(n,xs) = length xs sinkt dagegen bei jedem rekursiven Aufruf strikt, weil Pattern 3 die Liste immer um 1 kürzt.

9.🎓 Nachklausur 25.09.2025: Aufgabe 5a — Was macht repeatUntil p action, wenn p x für den ausgepackten Wert False liefert?

repeatUntil p action = do { x <- action; if p x then return () else repeatUntil p action }. Liefert p x False, ruft sich repeatUntil mit DERSELBEN Aktion erneut auf — dadurch wird action erneut ausgeführt und liefert (z. B. bei IO) einen frischen Wert.

10.🎓 Nachklausur 25.09.2025: Aufgabe 5b — Mit welcher Funktion liest askNumber die Nutzereingabe als String ein, bevor sie mit read in eine Zahl umgewandelt wird?

line <- getLine liest die Nutzereingabe als String ein — getLine hat den Typ IO String.
