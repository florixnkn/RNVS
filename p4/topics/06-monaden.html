Worum geht's?

Eine Monade ist eine Berechnung, die mit einem Kontext läuft — zum Beispiel „könnte fehlschlagen" (Maybe), „liefert mehrere Ergebnisse" ([]), „macht Seiteneffekte" (IO) oder „trägt einen eigenen Fehlertyp" (Result). Monaden erlauben es, solche Berechnungen sauber hintereinander zu schalten, ohne bei jedem Schritt selbst den Kontext zu prüfen.

Stell dir vor …

… du bestellt Pakete über eine Lieferkette. Jedes Paket steckt in einem Karton (dem Kontext). Der Karton kann leer sein (Nothing), mehrere Inhalte haben ([1,2,3]) oder eine Quittung enthalten, die man erst unterschreiben muss (IO). Eine Monade ist wie ein Förderband, das den Karton automatisch weiterreicht: Ist ein Karton leer (Nothing), läuft das ganze Förderband weiter, ohne die leere Schachtel zu öffnen — das Ergebnis bleibt leer. Öffnest du dagegen einen Just 5-Karton, bekommst du die 5 heraus und kannst sie direkt weiterverarbeiten.

Typklassen-Hierarchie

Monad baut auf zwei Vorstufen auf. Jede Stufe fügt eine neue Fähigkeit hinzu:

Functor — der Grundbaustein

fmap wendet eine normale Funktion auf den Wert innerhalb des Kontexts an. Der Kontext selbst (z. B. Just, Nothing, oder eine Liste) bleibt erhalten:

fmap (+1) (Just 5)  =  Just 6    -- Kontext bleibt Just
fmap (+1) Nothing   =  Nothing   -- Kontext bleibt Nothing
fmap (*2) [1,2,3]   =  [2,4,6]   -- Kontext bleibt Liste

Applicative — Funktion im Kontext

pure verpackt einen Wert in den Kontext. (<*>) wendet eine im Kontext steckende Funktion auf einen im Kontext steckenden Wert an:

pure 5          :: Maybe Int  =  Just 5

Just (+1) <*> Just 5    =  Just 6
Nothing   <*> Just 5    =  Nothing
Just (+1) <*> Nothing   =  Nothing

[(*2), (*3)] <*> [1,2]  =  [2,4,3,6]  -- alle Kombinationen
Gut zu wissen: Warum reicht Applicative nicht?

Applicative kann Werte unabhängig kombinieren — aber es kann die zweite Berechnung nicht vom Ergebnis der ersten abhängig machen. Beim Typecheck von Equal e1 e2 musst du erst e1 prüfen, und dann — abhängig davon — e2 vergleichen. Das ist sequenzierte Abhängigkeit → genau das kann nur (>>=).

Monad — Bind und do-Notation

Der Bind-Operator (>>=) nimmt einen Wert aus dem Kontext heraus und steckt das Ergebnis der Folgefunktion in einen neuen Kontext. Die do-Notation ist nur syntaktischer Zucker dafür:

Monad-Instanzen

Maybe-Monad — Fehler elegant weitergeben

Sobald irgendwo Nothing auftaucht, fließt es still durch alle weiteren Schritte durch — wie ein Domino-Effekt. Du musst nirgends manuell prüfen:

instance Monad Maybe where
  return x         = Just x
  Nothing  >>= _  = Nothing   -- Fehler propagiert sofort
  (Just x) >>= f  = f x       -- Wert auspacken, Funktion aufrufen

-- Beispiel mit do:
do { x <- Just 5; y <- Just 3; return (x + y) }  =  Just 8

do { x <- Just 5; y <- Nothing; return (x + y) }  =  Nothing
--                      ^--- ein Nothing → alles Nothing
Merksatz: <- bedeutet nicht =

x <- Just 5 bedeutet: „Pack den Wert aus der Schachtel aus und nenne ihn x". x ist danach 5 (eine Zahl), nicht Just 5 (die Schachtel). Genau das ist der Unterschied zu let x = Just 5.

List-Monad — alle Kombinationen

Bei Listen bedeutet >>=: jeden Wert der Liste durch die Funktion jagen und alle Ergebnisse zusammenkleben — concat . map:

instance Monad [] where
  return x   = [x]                   -- Wert in Einzel-Liste
  xs >>= f   = concat (map f xs)     -- alle Ergebnisse zusammenkleben

-- Alle Paare aus [1,2] × [3,4]:
do { x <- [1,2]; y <- [3,4]; return (x,y) }
= [1,2] >>= \x -> [3,4] >>= \y -> return (x,y)
= [(1,3),(1,4),(2,3),(2,4)]

Result-Monad — eigener Fehlertyp

Statt Nothing ohne Erklärung kann Result einen Fehlertext mitliefern:

data Result a = Success a | Failure String

instance Functor Result where
  fmap f (Success a) = Success (f a)
  fmap _ (Failure e) = Failure e         -- Fehler unverändert durchleiten

instance Applicative Result where
  pure = Success
  (Failure e) <*> _           = Failure e
  _           <*> (Failure e) = Failure e
  (Success f) <*> (Success a) = Success (f a)

instance Monad Result where
  return = Success
  (Failure e) >>= _ = Failure e         -- Fehler propagiert
  (Success a) >>= f = f a               -- Weiter mit dem Wert
Achtung: Reihenfolge der Instanzen

Du kannst Monad Result erst definieren, wenn sowohl Functor Result als auch Applicative Result bereits definiert sind. Haskell erzwingt die Hierarchie.

Monad-Gesetze

Jede korrekte Monad-Instanz muss drei Gesetze erfüllen:

-- 1. Left Identity: return dann bind = direkt die Funktion
return x >>= f   =  f x

-- 2. Right Identity: bind mit return = keine Änderung
m >>= return      =  m

-- 3. Assoziativität: Klammerung spielt keine Rolle
(m >>= f) >>= g  =  m >>= (\x -> f x >>= g)
Signatur von bind
(>>=) :: m a -> (a -> m b) -> m b

m a = Wert im Kontext · (a -> m b) = Funktion, die Wert nimmt und neuen Kontext liefert · m b = Ergebnis im Kontext.

Schritt für Schritt: Typecheck mit do-Notation

Das Klausurbeispiel aus ProMo'24 (Task 5): Equal e1 e2 soll typgeprüft werden. Zuerst den Typ von e1 bestimmen, dann abhängig davon den Typ von e2 prüfen und vergleichen — genau das braucht eine Monade:

// typecheck-equal.hs
typecheck :: Expr -> Maybe Type
typecheck (Equal e1 e2) = do
  t1 <- typecheck e1    -- Typ von e1 auspacken (oder Nothing)
  t2 <- typecheck e2    -- Typ von e2 auspacken (oder Nothing)
  if t1 == t2
    then Just TBool     -- gleiche Typen → Bool
    else Nothing        -- verschiedene Typen → Fehler
  1. Schritt 1 — t1 auspacken: typecheck e1 liefert z. B. Just TInt. Das <- packt aus: t1 = TInt.
  2. Schritt 2 — t2 auspacken: typecheck e2 liefert Just TInt. t2 = TInt.
  3. Schritt 3 — vergleichen: t1 == t2 ist True → Ergebnis Just TBool.
  4. Wenn ein Schritt Nothing liefert: Die Monade leitet es durch — alle weiteren Schritte werden übersprungen.
typecheck (Equal (ConstI 1) (ConstI 2)) → do { t1 <- Just TInt; t2 <- Just TInt; … }
→ Just TInt >>= \t1 -> Just TInt >>= \t2 -> if TInt==TInt then Just TBool else Nothing
→ if True then Just TBool else Nothing → Just TBool

Desugaring im Detail

Der Compiler wandelt das do mechanisch um:

-- do-Block:
do
  t1 <- typecheck e1
  t2 <- typecheck e2
  if t1 == t2 then Just TBool else Nothing

-- desugars zu:
typecheck e1 >>= \t1 ->
  typecheck e2 >>= \t2 ->
    if t1 == t2 then Just TBool else Nothing
Warum hier kein Applicative?

Mit liftA2 (Applicative) könnte man t1 und t2 unabhängig berechnen. Aber der Vergleich if t1 == t2 hängt vom Ergebnis beider ab — das ist eine Abhängigkeit zwischen Berechnungen. Nur (>>=) kann das ausdrücken, weil seine zweite Eingabe eine Funktion a -> m b ist (sie sieht den Wert a).

Typische Fallen (Klausur!)

❌ Falsch<- wie eine Gleichung lesen — x ist die Schachtel: 
do
  x <- Just 5
  return x        -- x wäre dann Just 5?!
✅ Richtig<- bedeutet „auspacken": x ist der Inhalt, nicht die Schachtel: 
-- x ist 5 (eine Zahl), nicht Just 5
-- return (x+1) = return 6 = Just 6
❌ FalschMonad-Instanz ohne Functor/Applicative definieren: 
instance Monad Result where
  return = Success
  (Failure e) >>= _ = Failure e
  (Success a) >>= f = f a
-- Compilerfehler: Functor und Applicative fehlen!
✅ RichtigImmer alle drei Instanzen in der richtigen Reihenfolge: 
instance Functor Result where ...
instance Applicative Result where ...
instance Monad Result where ...
❌ FalschFehler in bind nicht weiterleiten — Failure geht verloren: 
instance Monad Result where
  (Failure e) >>= f = f undefined  -- FALSCH!
✅ RichtigFehler immer mit dem Unterstrich-Pattern still durchleiten: 
instance Monad Result where
  (Failure e) >>= _ = Failure e   -- Fehler propagiert
  (Success a) >>= f = f a
❌ FalschEqual-Typecheck mit Applicative versuchen — geht nicht, da abhängig: 
-- liftA2 kann t1 == t2 NICHT prüfen!
typecheck (Equal e1 e2) =
  liftA2 (==) (typecheck e1) (typecheck e2)
✅ RichtigAbhängige Sequenzierung → Monad / do-Notation: 
typecheck (Equal e1 e2) = do
  t1 <- typecheck e1
  t2 <- typecheck e2
  if t1 == t2 then Just TBool else Nothing
❌ FalschList-Monad-Ergebnis falsch abschätzen — nicht summieren, sondern Paare zählen: 
-- [1,2] × [3,4] = 2+2 = 4? Oder 2·2 = 4?
-- Egal: nicht alle addieren!
✅ RichtigList-Monad erzeugt das kartesische Produkt — alle Kombinationen: 
do { x <- [1,2]; y <- [3,4]; return (x,y) }
= [(1,3),(1,4),(2,3),(2,4)]   -- 2 × 2 = 4 Paare

🧩 Wissenscheck

Drei kurze Fragen — tippe deine Antwort ein oder klick die richtige Option. Grün = richtig.

1.Was ergibt do { x <- Just 5; y <- Just 3; return (x + y) }?

Beide Werte sind Just, also wird ausgepackt: 5 + 3 = 8 → Just 8.

2.Welche Signatur gehört zu (>>=) (bind)?

Das ist bind. Die zweite Signatur ist fmap (Functor), die dritte ist (<*>) (Applicative).

3.List-Monade: Wie viele Paare enthält do { x <- [1,2]; y <- [3,4]; return (x, y) }?

Alle Kombinationen: (1,3),(1,4),(2,3),(2,4) → 4 Paare.
