Worum geht's?

UDP ist das schnelle, unkomplizierte Transportprotokoll im Internet. Wenn du Daten schnell vom einen Computer zum anderen schicken willst und es dir nicht ausmacht, wenn manchmal eine Nachricht verlorengeht, dann ist UDP dein Protokoll. Keine Anrufe vorher, keine Versprechen — einfach absenden, wie eine Postkarte.

Stell dir vor …

… du schickst deine Pizza-Bestellung als Postkarte zur Pizzeria statt per Telefon anzurufen. Die Postkarte ist schnell unterwegs, du musst nicht „Hallo?" sagen und auf eine Antwort warten. Aber es gibt auch Probleme: Wenn die Postkarte verloren geht, weiß die Pizzeria nicht, dass du bestellt hast. Wenn mehrere Postkarten ankommen, könnte sie dir fünf Pizzen liefern. Und wenn deine Adresse auf einer Postkarte fehlt, kriegt deine Pizza der Nachbar. Das ist UDP: schnell, aber ohne Garantien.

Was ist UDP?

UDP steht für User Datagram Protocol und ist ein Transportschicht-Protokoll (Schicht 4 im Internet-Modell). Es wurde 1980 in RFC 768 definiert — eine der kurz und knapp geschriebenen RFCs überhaupt!

Zentrales Merkmal: Verbindungslosigkeit

UDP ist verbindungslos. Das heißt: Es gibt kein „Hallo" vorher, keinen Handshake, keine Vereinbarung. Du packst deine Daten einfach ein und schickst sie los. Der Empfänger weiß erst Bescheid, wenn das erste Paket ankommt.

Gut zu wissen

UDP ist verbindungslos UND unzuverlässig — aber das ist manchmal genau das Richtige! Bei VoIP, Online-Spielen oder Streaming ist eine kurze Aussetzer besser als eine Verzögerung. UDP hat keine Wiederholung von verlorenen Paketen (das würde zu Lag führen).

UDP vs. TCP — Der Schnell-Vergleich

Merksatz

UDP = Best-Effort-Protokoll. Es versucht, deine Daten zu übertragen, aber garantiert nichts. Keine Verbindung, kein Handshake, keine Reihenfolge, keine Wiederholung. Dafür: minimal schnell, minimal Overhead. Perfekt für Anwendungen, die Echtzeit brauchen oder selbst Zuverlässigkeit handhaben (wie QUIC).

UDP-Header aufbauen — Schritt für Schritt

Wenn eine Anwendung ein UDP-Paket versendet, passieren genau diese Schritte:

  1. Source Port wählen: Meist ein zufälliger hoher Port (ephemer, >1023), damit der Empfänger antwortet kann.
  2. Destination Port bestimmen: Der Port, an den die Nachricht gehen soll (z.B. Port 53 für DNS).
  3. Length berechnen: Header (8 Byte) + Nutzdaten zusammenzählen.
  4. Checksum berechnen: Eine Prüfsumme über alle Daten (erkennt Bitfehler).
  5. Header zusammensetzen: Source Port | Dest Port | Length | Checksum — genau 8 Byte.
  6. An die IP-Schicht weitergeben: Die IP-Schicht schickt es dann ins Netzwerk.

Beispiel: sendto-Pseudocode analysieren

Dies ist echter Code aus einer Prüfung (Blatt 5). Schauen wir uns an, was hier passiert:

def sendto(bytes, (address, dst_port)):
    address_bytes = hton(address)           # IP-Adresse konvertieren
    destination_port = hton(dst_port)       # Ziel-Port
    source_port = hton(random())            # Zufälliger Sender-Port
    length = hton(len(bytes) + 8)           # 8 Byte Header + Nutzdaten
    checksum = build_checksum(...)          # Prüfsumme
    PDU = length + destination_port + source_port + checksum + bytes
    route_packet(address_bytes, PDU)        # An IP-Schicht
    return
Was zeigt dieser Code?

PDU-Aufbau: Header (4 Felder à 2 Byte = 8 Byte) + Nutzdaten.

Verbindungslos? Ja! Es gibt keinen State, kein Handshake, jeder Aufruf ist unabhängig.

Source Port ist zufällig: `random()` — das ist das Erkennungszeichen von UDP (TCP würde eine Sequence Number haben).

Length = Header + Daten: `len(bytes) + 8` — wichtig, oft Fehlerquelle!

Typische Fallen

❌ FalschUDP-Header ist 20 Byte groß.
✅ RichtigUDP-Header ist 8 Byte. TCP-Header ist 20 Byte (mindestens). Das ist ein wichtiger Unterschied!
❌ FalschUDP hat keine Checksum.
✅ RichtigUDP HAT eine Checksum (2 Byte) — aber nur zur Erkennung von Fehlern, nicht zur Behebung. Wenn die Checksum fehlschlägt, verwirft UDP das Paket einfach. TCP würde es neu anfordern.
❌ FalschUDP ist immer schlechter als TCP.
✅ RichtigUDP ist für Echtzeit-Anwendungen besser! Bei VoIP, Online-Spielen oder Live-Streaming ist eine kurze Aussetzer (ein verlorenes Paket) weit weniger schlimm als Verzögerung (durch TCP-Wiederholung). Schnelligkeit schlägt Zuverlässigkeit.
❌ FalschDie Length im UDP-Header zählt nur die Nutzdaten.
✅ RichtigDie Length zählt den gesamten UDP-Block: Header (8 Byte) + Nutzdaten. Im Pseudocode: `length = len(bytes) + 8`. Ein häufiger Fehler!
❌ FalschPizza über UDP: Alle Informationen (Name, Adresse, Pizza-Typ) kommen in der richtigen Reihenfolge an.
✅ RichtigUDP garantiert keine Reihenfolge. Deine Bestellung könnte ankommen als: „Pizza Salami", dann (Stunden später) „Adresse: Hauptstr. 5", dann „Name: Florian". Das ist chaotisch — deshalb ist UDP für Pizza ungeeignet!
❌ FalschSource Port in UDP ist immer 0 oder fix.
✅ RichtigSource Port ist meist ein zufälliger, ephemerer Port (>1023), damit der Empfänger weiß, wohin er antworten soll. Im Code: `random()` — das ist normal und wichtig!

Typische UDP-Anwendungen (und ihre Ports)

  • Port 53: DNS (Domain Name System)
  • Port 67/68: DHCP (dynamische IP-Vergabe)
  • Port 123: NTP (Zeitsynchronisation)
  • Port 69: TFTP (Trivial File Transfer)
  • Variable: VoIP, RTP (Real-time Transport)
  • Variable: Online-Games
  • Port 443: QUIC/HTTP/3 (Zuverlässigkeit in der App)

RFC 768 Kernpunkte

  1. Was beschreibt RFC 768? UDP — ein minimales Transportprotokoll mit „Minimum of Protocol Mechanism".
  2. Zentrales Merkmal? Verbindungslos + Best-Effort (keine Garantien).
  3. Wofür ideal? Echtzeit-Anwendungen, die Geschwindigkeit über Zuverlässigkeit stellen.
  4. Was braucht die Anwendung selbst? Wenn Zuverlässigkeit nötig ist (z.B. QUIC), bauen wir sie über UDP.
Merksatz für die Klausur

UDP = 8 Byte, verbindungslos, schnell, Best-Effort. Keine Wiederholung, keine Reihenfolge, keine Flusskontrolle — aber auch kein Overhead. Perfect für Echtzeitanwendungen wie VoIP und Online-Games. Wenn du Zuverlässigkeit brauchst → TCP oder Anwendungslogik.
