Worum geht's?

Algebraische Datentypen (ADTs) sind die Bausteine, mit denen du in Haskell völlig neue Datenstrukturen erfindet. Es gibt zwei Grundformen: Sum Types (entweder/oder, getrennt mit |) und Product Types (alles zusammen, mehrere Felder nach einem Konstruktor). Kombiniert ergeben sie beliebig komplexe Strukturen — von Listen über Bäume bis zu Navigations-Cursors.

Stell dir vor …

… ein Sum Type ist wie ein Lichtschalter: Er ist entweder AN oder AUS — nie beides gleichzeitig. Mit dem |-Zeichen listest du alle möglichen Zustände auf, und Haskell stellt sicher, dass immer genau einer davon gilt.

Ein Product Type hingegen ist wie ein Personalausweis: Er hat gleichzeitig einen Namen, ein Geburtsdatum und eine Ausweisnummer. Alle Felder existieren immer zusammen — daher der Name „Product" (Kreuzprodukt der Werte).

Theorie: ADTs

In Haskell definierst du neue Typen mit dem Schlüsselwort data. Was danach kommt, sind die Konstruktoren — Funktionen, die Werte dieses Typs erzeugen.

Sum Types vs. Product Types

-- Sum Type: ENTWEDER Empty ODER Pos (mit |)
data Zipper a = Empty | Pos [a] a [a]

-- Product Type: ALLE Felder zusammen
data Node = Node Int String Bool

Rekursive Datentypen

Ein Typ kann sich selbst als Feld enthalten — genau wie bei rekursiven Funktionen. So baut man in Haskell Listen und Bäume von Grund auf selbst:

-- Liste selbst gebaut
data List a = Nil | Cons a (List a)

-- Binärbaum
data Tree a = Leaf | Branch (Tree a) a (Tree a)

-- Peano-Zahlen
data Peano = Zero | Succ Peano

Peano-Zahlen sind besonders elegant: Statt eine Zahl direkt zu speichern, wird sie durch Schachtelung dargestellt. Die Zahl 3 ist Succ (Succ (Succ Zero)) — drei Schachteln, ganz unten liegt Zero.

Zipper — Navigation mit Fokus

Ein Zipper ist eine Datenstruktur, die eine Liste mit einem markierten Fokuspunkt darstellt. Du kannst dich durch die Liste bewegen, indem du den Fokus nach links oder rechts verschiebst — ohne die Elemente zu verlieren.

data Zipper a = Empty | Pos [a] a [a]
-- Pos linke_elemente fokus rechte_elemente
-- Invariante: gesamt = linke ++ [fokus] ++ rechte

moveRight :: Zipper a -> Maybe (Zipper a)
moveRight Empty            = Just Empty
moveRight (Pos ls c [])    = Nothing
moveRight (Pos ls c (r:rs)) = Just (Pos (ls ++ [c]) r rs)
Invariante des Zippers

Es gilt immer: gesamt = ls ++ [c] ++ rs. Das bedeutet: moveRight verschiebt nur den Fokus — kein Element geht verloren, kein Element wird doppelt gezählt. Das ist eine strukturelle Garantie des Typs.

Commands — Aktions-Sequenzen

Commands ist ein rekursiver ADT für eine Folge von Aktionen. Statt einer Liste von Aktionen hat man eine eigene Datenstruktur — und kann ihr eigenes Gleichheitskonzept definieren.

data Commands = Done | Then Action Commands
data Action = North | South | West | East

-- fromList: Liste zu Commands
fromList :: [Action] -> Commands
fromList []     = Done
fromList (x:xs) = Then x (fromList xs)

-- Kurzform via foldr:
fromList' :: [Action] -> Commands
fromList' = foldr Then Done

-- moveAll: Sequenz ausführen
moveAll :: (Int, Int) -> Commands -> (Int, Int)
moveAll pos Done        = pos
moveAll pos (Then a cs) = moveAll (move a pos) cs

-- Eq: gleiche Endposition = gleiche Commands
instance Eq Commands where
  c1 == c2 = moveAll (0,0) c1 == moveAll (0,0) c2
Eigenes Eq definieren

Zwei Commands-Werte können strukturell verschieden, aber semantisch gleich sein — z.B. Then North (Then South Done) und Done landen beide bei (0,0). Mit instance Eq können wir genau das ausdrücken.

Result — Fehlerbehandlung

data Result a = Success a | Failure String
Fehler als Werte, nicht als Exceptions

Statt Exceptions wirft Haskell keinen Fehler — der Fehler wird als Wert weitergegeben. Result ist Haskells eigene Variante von Either: Success a enthält das Ergebnis, Failure String die Fehlermeldung. Pattern Matching zwingt dich, beide Fälle zu behandeln.

Übersicht aller ADTs aus dem Blatt

ADT Konstruktoren Zweck

List a Nil, Cons a (List a) selbstgebaute Liste
Tree a Leaf, Branch (Tree a) a (Tree a) Binärbaum
Peano Zero, Succ Peano Natürliche Zahlen strukturell

Result a Success a, Failure String Fehlerbehandlung

Schritt für Schritt: moveRight und fromList

Wir rechnen zwei Klausur-typische Ausdrücke vollständig durch — genau so, wie du es aufschreiben sollst.

Beispiel 1: moveRight (Pos [1,2] 3 [4,5])

  1. Ausgangszustand: Wir haben einen Zipper mit linke=[1,2], fokus=3, rechte=[4,5].
  2. Pattern Matching: Das Muster Pos ls c (r:rs) passt mit ls=[1,2], c=3, r=4, rs=[5].
  3. Ergebnis bilden: Just (Pos ([1,2]++[3]) 4 [5]) = Just (Pos [1,2,3] 4 [5]).
moveRight (Pos [1,2] 3 [4,5]) → Just (Pos ([1,2]++[3]) 4 [5]) → Just (Pos [1,2,3] 4 [5])

Beispiel 2: fromList [North, East]

  1. Erster Schritt: fromList (North:xs) mit xs=[East] → Then North (fromList [East]).
  2. Zweiter Schritt: fromList (East:xs) mit xs=[] → Then East (fromList []).
  3. Base Case: fromList [] = Done — Rekursion endet!
  4. Zusammensetzen: Ergibt Then North (Then East Done).
fromList [North, East] → Then North (fromList [East]) → Then North (Then East Done)
Merksatz für rekursive ADTs

Bei rekursiven ADTs: immer alle Konstruktoren abdecken (jeden mit | getrennten Fall). Kein Fall darf fehlen — sonst gibt es zur Laufzeit einen Fehler. Und: Der rekursive Konstruktor (Cons, Then, Succ) enthält immer den Typ selbst — das erzeugt die Schachtelstruktur.

Typische Fallen (Klausur!)

❌ Falsch moveRight Empty gibt Nothing zurück — ein leerer Zipper kann sich ja nicht bewegen: 
moveRight Empty = Nothing
✅ Richtig Laut Spezifikation bleibt ein leerer Zipper leer — er gibt Just Empty zurück, keinen Fehler: 
moveRight Empty = Just Empty
❌ Falsch Kein Base Case für Peano — führt zu Endlosrekursion bei n=0: 
intToPeano n = Succ (intToPeano (n-1))
✅ Richtig Base Case für 0 zuerst, dann der rekursive Fall: 
intToPeano 0 = Zero
intToPeano n = Succ (intToPeano (n-1))
❌ Falsch fromList einfach als Identität — Type Error, weil Commands keine Liste ist: 
fromList xs = xs
✅ Richtig Jedes Element mit Then umhüllen, Base Case mit Done: 
fromList []     = Done
fromList (x:xs) = Then x (fromList xs)
❌ Falsch moveAll ohne Base Case — Done wird nie behandelt, Programmabsturz: 
moveAll pos (Then a cs) = moveAll (move a pos) cs
✅ Richtig Beide Konstruktoren von Commands abdecken: 
moveAll pos Done        = pos
moveAll pos (Then a cs) = moveAll (move a pos) cs

🎯 Quick-Reference für die Klausur

Muster Konstruktor Signatur / Beispiel
Sum Type data X = A | B entweder A oder B
Product Type data X = X Int String alle Felder zusammen
Rekursiver Typ Nil / Cons data List a = Nil | Cons a (List a)
Zipper-Navigation moveRight Pos ls c (r:rs) -> Just (Pos (ls++[c]) r rs)
Aktionssequenz Then / Done fromList = foldr Then Done
Peano-Zahl Zero / Succ intToPeano 0 = Zero; intToPeano n = Succ (intToPeano (n-1))

Jedes data-Statement definiert neue Konstruktoren. Sum Types mit | trennen die Fälle. Product Types haben mehrere Felder nach dem Konstruktornamen. Bei Pattern Matching immer alle Konstruktoren abdecken — die Klausur prüft genau das.

🧩 Wissenscheck

Drei kurze Fragen — tippe deine Antwort ein oder klick die richtige Option. Grün = richtig.

1.Was ergibt fromList [North, East] (Umwandlung einer Liste in Commands)?

fromList (x:xs) = Then x (fromList xs), also zuerst Then North ..., dann Then East Done.

2.Was liefert moveRight Empty beim Zipper?

Ein leerer Zipper ist bereits „am Ende" — laut Definition moveRight Empty = Just Empty (nicht Nothing).

3.Was ist intToPeano 2?

intToPeano 0 = Zero, sonst Succ (intToPeano (n-1)) ⟶ Succ (Succ Zero).
