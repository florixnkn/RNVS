Aufgabe 1 (10 Punkte): Rekursion — Vektorbetrag

Original-Aufgabentext: Der Betrag eines Vektors x = (x₁,…,xₙ) ist die Quadratwurzel aus der Summe der Quadrate seiner Komponenten. Der Vektor wird als nicht-leere endliche Liste dargestellt.

Formel
|x| = √(x₁² + x₂² + … + xₙ²)

(a) 2 P. — magnitude, rekursiv (nicht endrekursiv)

Original-Aufgabentext: Vervollständige magnitude mit einer rekursiven, aber nicht endrekursiven Hilfsfunktion sumOfSquares.

magnitude :: Floating a => [a] -> a
magnitude = sqrt . sumOfSquares
  where sumOfSquares [] = ...
        sumOfSquares (x:xs) = ...

Musterlösung:

magnitude :: Floating a => [a] -> a
magnitude = sqrt . sumOfSquares
  where sumOfSquares [] = 0
        sumOfSquares (x:xs) = x * x + sumOfSquares xs

Der Basisfall liefert 0 (neutrales Element der Addition). Im rekursiven Fall wird das Quadrat von x zum rekursiven Ergebnis addiert — nicht x selbst, sonst wäre es die Summe statt der Quadratsumme.

(b) 4 P. — magnitude', endrekursiv

Original-Aufgabentext: Formuliere die Funktion so, dass die Vektorsumme endrekursiv ausgedrückt wird. magnitude' soll sich ansonsten identisch zu magnitude verhalten.

Musterlösung:

magnitude' :: Floating a => [a] -> a
magnitude' xs = sqrt (go xs 0)
  where go [] acc = acc
        go (x:xs) acc = go xs (x * x + acc)

Die Hilfsfunktion braucht einen Akkumulator als zusätzliches Argument. Der Rumpf von magnitude' kann nicht mehr punktfrei (sqrt . go) geschrieben werden, weil ein Startwert (0) für den Akkumulator übergeben werden muss. Entscheidend für Endrekursion: x * x wird vor dem rekursiven Aufruf in den Akkumulator eingerechnet (go xs (x*x + acc)) — nach dem rekursiven Aufruf passiert nichts mehr.

(c) 4 P. — magnitude'', mit foldr

Original-Aufgabentext: Definiere magnitude'' ohne explizite Rekursion, mithilfe der Listenfunktion foldr.

Musterlösung:

magnitude'' :: Floating a => [a] -> a
magnitude'' xs = sqrt (foldr (\x acc -> x * x + acc) 0 xs)

Vergleiche die Struktur von sumOfSquares aus (a) mit der Definition von foldr selbst (foldr f z [] = z, foldr f z (x:xs) = f x (foldr f z xs)) — beide sehen fast identisch aus. Der Basisfall [] = 0 wird zum Startwert z = 0, und der rekursive Fall x*x + sumOfSquares xs wird zur Kombinationsfunktion f x acc = x*x + acc.

❌ FalschBei (a)/(c) vergessen, x zu quadrieren: 
sumOfSquares (x:xs) = x + sumOfSquares xs   -- FALSCH: das ist nur die Summe!
✅ Richtigx mit sich selbst multiplizieren, bevor addiert wird: 
sumOfSquares (x:xs) = x * x + sumOfSquares xs
❌ FalschBei (b) die Addition NACH dem rekursiven Aufruf stehen lassen — das ist gar keine Endrekursion mehr: 
go (x:xs) acc = x * x + go xs acc   -- FALSCH: nicht endrekursiv!
-- Nach dem rekursiven Aufruf muss noch "x*x + ..." berechnet werden.
✅ Richtigx*x VOR dem rekursiven Aufruf in den Akkumulator einrechnen: 
go (x:xs) acc = go xs (x * x + acc)

Aufgabe 2 (10 Punkte): Polymorphie — Binärbaum mit Foldable

data Tree a = Empty | Leaf a | Node (Tree a) (Tree a)

Original-Aufgabentext: Ein Binärbaum, der Knotenmarkierungen nur in Blättern (Leaf) trägt, aber auch leere Teilbäume enthalten kann (Empty).

(a) 4 P. — findValue :: Eq a => a -> Tree a -> Bool

Original-Aufgabentext: Vervollständige die Funktion, die prüft, ob ein Wert im Baum vorkommt.

findValue :: Eq a => a -> Tree a -> Bool
findValue v Empty = False
findValue v (Leaf a) = v == a
findValue v (Node l r) = findValue v l || findValue v r

Empty enthält nie den gesuchten Wert (False). Bei Leaf a wird direkt verglichen. Bei Node l r reicht es, wenn der Wert in einem der beiden Teilbäume vorkommt — daher ||, nicht &&.

(b) 4 P. — Foldable-Instanz

Original-Aufgabentext: Definiere für Tree a eine sinnvolle Instanz der Typklasse Foldable über foldMap :: Monoid m => (a -> m) -> Tree a -> m.

instance Foldable Tree where
  foldMap f Empty = mempty
  foldMap f (Leaf a) = f a
  foldMap f (Node l r) = foldMap f l <> foldMap f r

Genau wie bei findValue: Empty trägt nichts bei (mempty), Leaf a wendet f auf den einen Wert an, und bei Node werden die Beiträge beider Teilbäume mit <> zusammengeführt.

(c) 2 P. — allEven mit foldr

Original-Aufgabentext: Nutze, dass Tree a Foldable ist, und definiere allEven :: Integral a => Tree a -> Bool mithilfe von foldr.

allEven :: Integral a => Tree a -> Bool
allEven = foldr (\x acc -> even x && acc) True

Startwert True (neutrales Element von &&) — ein leerer Baum hat „alle" Elemente gerade, vacuously true. Jedes Element muss gerade sein und der Rest des Baumes auch, daher &&.

❌ FalschBei (a) && statt || verwenden: 
findValue v (Node l r) = findValue v l && findValue v r
-- FALSCH: würde verlangen, dass der Wert in BEIDEN Teilbäumen vorkommt!
✅ RichtigEin Vorkommen in irgendeinem Teilbaum genügt: 
findValue v (Node l r) = findValue v l || findValue v r
❌ FalschBei (c) mit False statt True starten: 
allEven = foldr (\x acc -> even x && acc) False
-- FALSCH: liefert für JEDEN Baum False, weil False && irgendwas immer False bleibt!
✅ RichtigNeutrales Element von && als Startwert: 
allEven = foldr (\x acc -> even x && acc) True

Aufgabe 3 (10 Punkte): Datentypen — Bundesliga-Tabelle

Original-Aufgabentext: Die Bundesliga-Tabelle soll als verknüpfte Liste dargestellt werden: jeder Knoten speichert Mannschaftsname (String), Punktzahl (Int) und einen Verweis auf den nächsten Knoten. Das Ende wird mit Nil markiert.

(a) 2 P. — data Bundesliga

Original-Aufgabentext: Definiere den rekursiven Datentyp mit den Konstruktoren Position und Nil — ohne eingebaute Datentypen wie Listen oder Tupel.

data Bundesliga = Position String Int Bundesliga
                 | Nil

Position nimmt genau die drei in der Aufgabe genannten Felder als eigene Argumente (nicht als Tupel gebündelt) — das dritte Feld ist wieder vom Typ Bundesliga selbst, das macht den Typ rekursiv.

(b) 4 P. — inTable

Original-Aufgabentext: Definiere inTable :: String -> Bundesliga -> Bool, das prüft, ob eine Mannschaft in der Tabelle vorkommt.

inTable :: String -> Bundesliga -> Bool
inTable _ Nil = False
inTable name (Position n _ rest)
  | name == n = True
  | otherwise = inTable name rest

Basisfall Nil: die leere Tabelle enthält keine Mannschaft. Sonst wird der Name des aktuellen Knotens verglichen; bei Treffer True, sonst rekursiv im Rest der Tabelle weitersuchen.

(c) 4 P. — inOrder

Original-Aufgabentext: Definiere inOrder :: Bundesliga -> Bool, das prüft, ob die Tabelle absteigend sortiert ist (jedes Team ≥ Punkte des nächsten Teams).

inOrder :: Bundesliga -> Bool
inOrder Nil = True
inOrder (Position _ _ Nil) = True
inOrder (Position _ p1 rest@(Position _ p2 _)) = p1 >= p2 && inOrder rest

Zwei Basisfälle: die leere Tabelle und eine Tabelle mit nur einem Eintrag (letzter Knoten vor Nil) sind trivial sortiert. Bei mindestens zwei Einträgen wird der Punktevergleich mit >= (nicht >, denn Gleichstand ist laut Aufgabe erlaubt) durchgeführt und rekursiv mit dem Rest der Tabelle fortgesetzt.

❌ FalschBei (a) ein eingebautes Tupel statt separater Felder verwenden: 
data Bundesliga = Position (String, Int) Bundesliga | Nil
-- FALSCH: Aufgabe verbietet explizit Tupel/Listen als Hilfsmittel
✅ RichtigDrei eigenständige Konstruktor-Argumente: 
data Bundesliga = Position String Int Bundesliga | Nil
❌ FalschBei (c) den Basisfall mit genau einem verbleibenden Eintrag vergessen: 
inOrder Nil = True
inOrder (Position _ p1 rest@(Position _ p2 _)) = p1 >= p2 && inOrder rest
-- FALSCH: Non-exhaustive patterns bei einer Tabelle mit genau einem Team!
✅ RichtigAuch den Ein-Element-Fall abdecken: 
inOrder (Position _ _ Nil) = True   -- letzter Eintrag: nichts mehr zu vergleichen

Aufgabe 4 (10 Punkte): Auswertung & Termination

(a) 2 P. — Call by Value, nächster Schritt

Original-Aufgabentext: Führe bei folgendem Ausdruck den nächsten Auswertungsschritt gemäß Call by Value durch.

(\x -> if x <= 0 then 1 else x * (x - 1)) (10 + 1)

Musterlösung: Call by Value wertet das Argument zuerst vollständig aus, bevor substituiert wird:

(\x -> if x <= 0 then 1 else x * (x - 1)) 11

(b) 2 P. — Call by Name, nächster Schritt

Original-Aufgabentext: Führe bei folgendem Ausdruck den nächsten Auswertungsschritt gemäß Call by Name durch.

(\y -> if y <= 1 then y * y * y else y) (10 + 1)

Musterlösung: Call by Name substituiert das Argument unausgewertet (an jeder Stelle, an der y vorkommt) in den Rumpf:

if (10 + 1) <= 1 then (10 + 1) * (10 + 1) * (10 + 1) else (10 + 1)
❌ FalschBei (b) das Argument trotzdem zuerst auswerten: 
if 11 <= 1 then 11 * 11 * 11 else 11
-- FALSCH: das wäre Call by Value, nicht Call by Name!
✅ RichtigBei Call by Name wird zunächst nur substituiert, nicht ausgewertet: 
if (10 + 1) <= 1 then (10 + 1) * (10 + 1) * (10 + 1) else (10 + 1)

(c) 6 P. — Terminationsbeweis für _sum

Original-Aufgabentext: Beweise, dass _sum für jede mögliche endliche Liste vom Typ [Int] terminiert.

sum :: [Int] -> Int
sum xs = _sum 0 xs where
  _sum akk [] = akk
  _sum akk (x:xs) = _sum (akk + x) xs

Musterlösung: Beweis durch (starke) Induktion über die Länge von xs.

  • Basisfall (xs = []): _sum akk [] = akk — kein rekursiver Aufruf, terminiert sofort.
  • Induktionsschritt (xs = y:ys): _sum akk (y:ys) = _sum (akk+y) ys. Die Liste ys ist echt kürzer als y:ys. Nach Induktionsvoraussetzung terminiert _sum (akk+y) ys, da ys eine kürzere endliche Liste ist. Also terminiert auch _sum akk (y:ys).

Formal als Abstiegsfunktion: m(akk, xs) = length xs ∈ ℕ₀. Bei jedem rekursiven Aufruf gilt m(akk+x, xs) = length xs < length (x:xs) = m(akk, x:xs). Da ℕ₀ wohlfundiert ist (keine unendlich absteigende Kette natürlicher Zahlen), terminiert _sum für jede endliche Liste.

Gut zu wissen

Dieselbe Beweisstruktur (Abgeschlossenheit, Definiertheit, Abstiegsfunktion) findet sich ausführlicher vorgeführt in der Nachklausur-Aufgabe zu skip — dort wird das Verfahren formaler in drei benannte Teilschritte zerlegt.

Aufgabe 5 (20 Punkte): Monaden — Variablen in Ausdrücken ersetzen

data Expr = Const Int
          | Var String
          | Plus Expr Expr
          deriving Show

type Env = [(String, Int)]

getVar :: Env -> String -> Maybe Int
getVar [] _ = Nothing
getVar ((k, val):xs) var | k == var  = Just val
                         | otherwise = getVar xs var

(a) 6 P. — replaceAp, applikativ

Original-Aufgabentext: Schreibe mithilfe von getVar und den Functor-/Applicative-Instanzen von Maybe eine applikative Funktion replaceAp, die Variablen durch Konstanten entsprechend ihrer Werte in Env ersetzt. Undefinierte Variable → Nothing für den ganzen Ausdruck.

ghci> replaceAp [("x", 1), ("y", 3)] (Var "x" `Plus` (Const 2 `Plus` Var "y"))
Just (Plus (Const 1) (Plus (Const 2) (Const 3)))
ghci> replaceAp [("x", 1)] (Var "x" `Plus` Var "y")
Nothing

Musterlösung:

replaceAp :: Env -> Expr -> Maybe Expr
replaceAp env (Const n)  = Just (Const n)
replaceAp env (Var c)    = Const <$> getVar env c
replaceAp env (Plus a b) = Plus <$> replaceAp env a <*> replaceAp env b

Eine Konstante bleibt unverändert. Eine Variable wird mit getVar nachgeschlagen (Maybe Int) und das Ergebnis mit Const <$> in einen Maybe Expr gehoben — schlägt die Suche fehl, bleibt es Nothing. Bei Plus a b müssen beide Teilausdrücke ersetzt werden; mit <*> werden zwei unabhängige Maybe-Werte kombiniert — ist einer der beiden Nothing, ist das Gesamtergebnis Nothing.

(b) 6 P. — replaceBd, mit return/>>=

Original-Aufgabentext: Schreibe eine zu replaceAp äquivalente Funktion replaceBd unter Verwendung von return und Bind (>>=) — nicht replaceAp selbst verwenden.

replaceBd :: Env -> Expr -> Maybe Expr
replaceBd env (Const n)  = return (Const n)
replaceBd env (Var c)    = getVar env c >>= \v -> return (Const v)
replaceBd env (Plus a b) =
  replaceBd env a >>= \a' ->
  replaceBd env b >>= \b' ->
  return (Plus a' b')

Bei Plus müssen die beiden rekursiven Ergebnisse sequenziell ausgepackt werden: erst a' aus replaceBd env a, dann — nur falls das erfolgreich war — b' aus replaceBd env b. Schlägt einer der beiden Binds fehl (Nothing), bricht die ganze Kette automatisch ab (das ist eingebautes Verhalten der Maybe-Monade).

(c) 2 P. — Warum keine Functor-Instanz für Expr?

Original-Aufgabentext: Wieso kann man für Expr keine Functor-Instanz definieren?

Musterlösung: Expr ist nicht über einen Typparameter polymorph — sein Kind ist *, nicht * -> *. Functor verlangt aber fmap :: (a -> b) -> f a -> f b, also einen Typkonstruktor f mit einem „Slot" für einen Elementtyp. Expr hat keinen solchen Parameter (die enthaltenen Werte sind fest auf Int bzw. String verdrahtet) — es gibt kein a, über das fmap abbilden könnte.

(d) 6 P. — buildEnv :: IO Env

Original-Aufgabentext: Lies eine Umgebung von der Standardeingabe: abwechselnd Variablenname und Wert in einer Zeile, bis der Benutzer den Variablennamen leer lässt.

buildEnv :: IO Env
buildEnv = do
  var <- getLine
  val <- getLine
  ...

Musterlösung:

buildEnv :: IO Env
buildEnv = do
  var <- getLine
  if null var
    then return []
    else do
      val <- getLine
      rest <- buildEnv
      return ((var, read val) : rest)

Die Abbruchbedingung (null var) muss direkt nach dem Einlesen von var geprüft werden — bevor überhaupt versucht wird, val einzulesen, sonst würde bei leerer Eingabe auf eine Wertzeile gewartet, die nie kommt. Im Erfolgsfall wird rekursiv der Rest der Umgebung eingelesen (buildEnv wieder von vorn) und das neue Paar vorne angehängt.

❌ FalschBei (a) die Variable-Konstruktor Var beibehalten, statt sie durch Const zu ersetzen: 
replaceAp env (Var c) = Var <$> getVar env c   -- FALSCH: Typfehler + falsche Semantik!
-- getVar env c :: Maybe Int, aber Var erwartet String, nicht Int
✅ RichtigDie gefundene Zahl in einen Const verpacken: 
replaceAp env (Var c) = Const <$> getVar env c
❌ FalschBei (d) die leere-Zeile-Prüfung vergessen — das Programm liest endlos weiter und crasht irgendwann bei getLine auf EOF: 
buildEnv = do
  var <- getLine
  val <- getLine
  rest <- buildEnv
  return ((var, read val) : rest)   -- FALSCH: kein Abbruch!
✅ RichtigDirekt nach var <- getLine auf null var prüfen und dann abbrechen: 
buildEnv = do
  var <- getLine
  if null var then return [] else do
    val <- getLine
    rest <- buildEnv
    return ((var, read val) : rest)

Merksatz für die Klausur

Bei Foldable-Instanzen (Tree, DT, …) immer alle Teilbäume mit <> zusammenführen — keinen vergessen. Bei Startwerten für foldr das neutrale Element der verwendeten Operation wählen (0 für +, True für &&, [] für ++). Bei eigenen ADTs keine eingebauten Typen verwenden, wenn die Aufgabe das explizit verbietet. Bei IO-Schleifen die Abbruchbedingung immer sofort nach dem Einlesen prüfen, bevor weitere Eingaben verlangt werden.

🧩 Wissenscheck

Alle Aufgaben stammen 1:1 aus der Probeklausur ProMo'24 (08.07.2024). Tippe deine Antwort ein oder wähle eine Option — du bekommst sofort Rückmeldung. Mit „Lösung" siehst du die Erklärung.

1.🎓 Probeklausur 08.07.2024: Aufgabe 1a — Gegeben magnitude = sqrt . sumOfSquares mit sumOfSquares (x:xs) = x*x + sumOfSquares xs. Was liefert magnitude [3, 4]?

sumOfSquares [3,4] = 3*3 + (4*4 + 0) = 9 + 16 = 25, sqrt 25 = 5 — das klassische 3-4-5-Dreieck.

2.🎓 Probeklausur 08.07.2024: Aufgabe 1b — Warum ist go (x:xs) acc = x * x + go xs acc KEINE gültige endrekursive Lösung?

Endrekursion heißt: der rekursive Aufruf ist die LETZTE Aktion. Hier muss nach go xs acc noch x*x + … berechnet werden — richtig wäre go xs (x*x + acc), wo die Addition VOR dem rekursiven Aufruf passiert.

3.🎓 Probeklausur 08.07.2024: Aufgabe 2a — Für data Tree a = Empty | Leaf a | Node (Tree a) (Tree a): welcher Operator gehört in findValue v (Node l r) = findValue v l ? findValue v r?

Der Wert muss nur in EINEM der beiden Teilbäume vorkommen, damit findValue True liefert — das ist ||, nicht && (das würde Vorkommen in BEIDEN Teilbäumen verlangen).

4.🎓 Probeklausur 08.07.2024: Aufgabe 2c — Für allEven = foldr (\x acc -> even x && acc) True: was liefert allEven Empty (leerer Baum)?

Ein leerer Baum enthält foldMap-mäßig keine Elemente, also greift direkt der Startwert True — „alle Elemente sind gerade" ist für eine leere Menge von Elementen vacuously true.

5.🎓 Probeklausur 08.07.2024: Aufgabe 3a — Wie viele Konstruktoren hat data Bundesliga = Position String Int Bundesliga | Nil?

2 Konstruktoren: Position (mit drei Feldern) und Nil (ohne Felder, markiert das Ende der Tabelle).

6.🎓 Probeklausur 08.07.2024: Aufgabe 3c — Warum braucht inOrder neben Nil noch einen zweiten Basisfall inOrder (Position _ _ Nil) = True?

Der dritte Fall Position _ p1 rest@(Position _ p2 _) matcht nur, wenn NOCH ein zweiter Position-Knoten folgt. Bei genau einem verbleibenden Eintrag (rest = Nil) würde ohne den zweiten Basisfall kein Pattern mehr passen.

7.🎓 Probeklausur 08.07.2024: Aufgabe 4a/b — Welche Auswertungsstrategie wertet (10 + 1) SOFORT zu 11 aus, bevor es in den Funktionskörper eingesetzt wird?

Call by Value wertet Argumente immer VOR der Substitution vollständig aus. Call by Name substituiert erst unausgewertet und wertet erst bei Bedarf im Rumpf aus.

8.🎓 Probeklausur 08.07.2024: Aufgabe 5c — Wieso kann man für data Expr = Const Int | Var String | Plus Expr Expr keine Functor-Instanz schreiben?

fmap :: (a -> b) -> f a -> f b braucht einen Typkonstruktor f mit einem Elementtyp-Parameter. Expr ist aber ein konkreter Typ (Kind *) ohne einen solchen Parameter — die Rekursion allein ist dabei kein Problem (andere rekursive Typen wie Tree a HABEN durchaus einen Parameter und damit eine Functor-Instanz).

9.🎓 Probeklausur 08.07.2024: Aufgabe 5d — Mit welcher Bedingung prüft buildEnv, ob die Eingabe beendet werden soll?

Direkt nach var <- getLine wird mit null var geprüft, ob die eingelesene Zeile leer ist — falls ja, wird sofort mit return [] abgebrochen, ohne eine Wertzeile einzulesen.
